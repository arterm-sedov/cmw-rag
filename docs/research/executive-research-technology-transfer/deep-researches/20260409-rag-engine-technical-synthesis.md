# Технический синтез: Архитектура и Know-How RAG-движка Comindware

## 1. Платформа и Агентная Схема
Интеграция с Comindware Platform строится на базе YAML-конфигурации (`rag_engine/config/cmw_platform.yaml`), которая является «единым источником правды».
- **Входная схема:** Описывает шаблон (`TPAIModel`) и поля (`title`, `question`, `browser`), запрашиваемые из платформы.
- **Шаблон запроса:** Формат Markdown, объединяющий поля ввода и системные метаданные.
- **Маппинг вывода:** Шаблон `agent_responses` сопоставляет внутренние состояния агента (`answer_confidence`, `diagnostics.guard.is_safe`) с атрибутами платформы.
- **Обработчики полей:** `mapping.py` преобразует массивы агента в HTML-списки для интерфейса платформы.

## 2. ETL и Индексация
Процесс индексации оптимизирован для инкрементальных обновлений.
- **Трехуровневое логирование времени:** Frontmatter → Git → FS. Система сравнивает `mtime` файла с `file_mtime_epoch` в векторном хранилище, пропуская неизмененные документы.
- **Стабильные Chunk IDs:** Глобально уникальные, но стабильные ID на базе сида документа, индекса чанка и хеша контента.
- **Режимы обработки:** `folder`, `file`, `mkdocs`. `RAGIndexer` координирует нарезку, эмбеддинг и пакетную запись.

## 3. Стек поиска и гибридная логика
Архитектура поиска использует многоступенчатый каскад.
- **Multi-Query и сегментация:** Запросы разбиваются на токены и опционально декомпозируются LLM на подзапросы для параллельного поиска.
- **Hybrid Search:** Векторный поиск по чанкам + реранкинг (cross-encoder или llm_reranker, например, DiTy, BGE, Qwen3).
- **Metadata Boosting:** Динамическая корректировка скоров на базе сигналов `tag_match`, `code_presence`, `section_match`.
- **Загрузка статей:** В отличие от стандартного RAG, возвращающего только чанки, движок загружает **полный контент статьи** из ФС для обеспечения LLM глубокого контекста.

## 4. Инференс и Управление Контекстом
Поддержка широкого спектра провайдеров (vLLM, Mosec, OpenRouter, Gemini, OpenAI).
- **Динамическое управление:**
  - Точная оценка токенов для предотвращения переполнения контекста.
  - Сжатие памяти: автоматическая суммаризация старых реплик.
  - Fallback-логика: автоматическое переключение на более мощную модель при нехватке окна контекста.

## 5. Наблюдаемость и Безопасность
- **Arize Phoenix:** Self-hosted стек на базе OpenInference/OpenTelemetry. Полная трассировка шагов, визуализация эмбеддингов и оценка качества (online/offline) без внешних облаков.
- **Qwen Guardian:** Контент-модерация на базе моделей Qwen3Guard (0.6B - 8B). Классификация по категориям риска (токсичность, предвзятость), статус выводится в платформу.

## 6. Асинхронное Взаимодействие
Агент работает по принципу «fire-and-forget» через `PlatformConnector`.
- **Workflow:** Платформа дергает агента -> коннектор забирает запись -> возвращает статус «started» -> RAG-пайплайн запускается в фоне -> агент сам создает запись ответа в платформе.

## 7. Требования к Железу (Сайзинг)
- **Minimum:** NVIDIA RTX 3060 (12GB VRAM).
- **Standard:** NVIDIA RTX 4090 (24GB). Достаточно для стека (Embedder + Reranker + Guardian 0.6B).
- **Performance:** NVIDIA RTX PRO 6000 (96GB) или A100 (80GB) для больших моделей (Qwen 72B).
- **Optimization:** Использование квантования (Q4/BF16) для снижения VRAM в 4 раза.
