## 1. OWASP LLM Top 10 2025: Валидация списка

**Официальный источник:** _«[OWASP Top 10 for LLM Applications 2025](https://genai.owasp.org/resource/owasp-top-10-for-llm-applications-2025/)»_ (опубликовано ноябрь 2024)

### Подтверждённый список

| # | Код | Категория | Краткое описание |
|---|------|----------|------------------|
| 1 | LLM01:2025 | Prompt Injection | Вредоносный ввод манипулирует поведением LLM; включает прямые и косвенные (сохранённые) инъекции |
| 2 | LLM02:2025 | Sensitive Information Disclosure | Модель раскрывает конфиденциальные данные, ПДн, учётные данные или ИС через вывод |
| 3 | LLM03:2025 | Supply Chain Vulnerabilities | Скомпрометированные модели, наборы данных, зависимости или сторонние сервисы |
| 4 | LLM04:2025 | Data and Model Poisoning | Обучающие данные или RAG-корпуса манипулируются для внедрения бэкдоров или предвзятости |
| 5 | LLM05:2025 | Improper Output Handling | Несанированный вывод LLM передаётся напрямую в системы, API или выполнение кода |
| 6 | LLM06:2025 | Excessive Agency | LLM предоставлены слишком большие полномочия/разрешения, позволяющие нежелательные вредоносные действия |
| 7 | LLM07:2025 | System Prompt Leakage | Несанкционированное раскрытие базовых системных инструкций и операционной логики |
| 8 | LLM08:2025 | Vector and Embedding Weaknesses | Уязвимости в механизмах поиска, пространствах эмбеддингов или векторных базах данных |
| 9 | LLM09:2025 | Misinformation | Генерация ложного или вводящего в заблуждение контента с риском чрезмерного доверия пользователей |
| 10 | LLM10:2025 | Unbounded Consumption | Чрезмерное потребление ресурсов, приводящее к DoS или атакам «отказ в кошельке» |

**Вывод:** Список в приложении D корректен. Отличие от версии 2023/24:
- LLM02 поднялся с #6 на #2 — раскрытие конфиденциальной информации теперь второй по значимости риск
- LLM07 (System Prompt Leakage) — новая категория
- LLM08 (Vector/Embedding Weaknesses) — новая категория для RAG-специфичных векторов атак
- LLM10 расширен — переименован из "Model Denial of Service" в "Unbounded Consumption"

## 4. Требования 152-FZ: Валидация

### Текущий статус (март 2026)

**Подтверждённые изменения:**

| Изменение | Дата вступления | Описание |
|-----------|-----------------|----------|
| Федеральный закон № 23-ФЗ | 1 июля 2025 | Ужесточение требований к локализации |
| Федеральный закон № 23-ФЗ | 1 сентября 2025 | Требования к анонимизации |
| Федеральный закон № 233-ФЗ | 8 августа 2024 | Экспериментальный правовой режим для ИИ |
| Проект закона о суверенном ИИ | Март 2026 | Минцифры опубликовало проект |

### Ключевые требования локализации (152-FZ, ст. 18)

**Действует с 1 июля 2025:** Часть 5 статьи 18 устанавливает, что при сборе персональных данных (включая через интернет) операторы **обязаны обеспечить** выполнение следующих операций с использованием баз данных, расположенных в России:
- Запись
- Систематизация
- Накопление
- Хранение
- Уточнение (обновление, изменение)
- Извлечение

**Запрещено:** Выполнение этих операций с использованием баз данных за пределами России для персональных данных российских граждан.

### Требования к анонимизации (действуют с 1 сентября 2025)

- **Необратимость:** Анонимизация должна быть необратимой
- **Разделение:** Анонимизированные данные не должны храниться вместе с оригинальными
- **Конфиденциальность методов:** Раскрытие методов анонимизации третьим лицам запрещено

### Проект закона о суверенном ИИ (март 2026)

Минцифры опубликовало проект федерального закона "Об основах государственного регулирования сфер применения технологий искусственного интеллекта":
- Нейросети должны создаваться в России и обучаться на российских данных
- Классификация моделей
- Контроль сервисов
- Маркировка контента

**Вывод:** Данные 152-FZ в документе корректны. Рекомендуется добавить информацию о проекте закона о суверенном ИИ (март 2026).

## 5. Инструменты observability: Валидация рекомендаций

### Статус OpenTelemetry GenAI Semantic Conventions

**Статус:** Development (не стабильно)

Текущие возможности:
- **Spans:** Стандартные атрибуты для LLM-инференса, эмбеддингов и операций поиска
- **Metrics:** Стандартизированные метрики использования токенов, задержек и стоимости
- **Events:** Событийная телеметрия для стриминга и захвата контента

**Миграция:** Существующие инструментации должны использовать переменную окружения `OTEL_SEMCONV_STABILITY_OPT_IN` со значением `gen_ai_latest_experimental`.

### Сравнение инструментов

| Инструмент | Тип | Self-hosted | 152-FZ совместимость | Статус (2026) |
|------------|------|-------------|----------------------|----------------|
| **Arize Phoenix** | OSS | ✅ Docker/K8s | ✅ Полное | Активен, 9k+ звёзд |
| **Langfuse** | OSS | ✅ Docker/K8s | ✅ Полное | Активен, v3 |
| **Helicone** | OSS proxy | ✅ Docker | ✅ Полное | Активен, 5,342 звёзды |
| **SigNoz** | OSS (full-stack) | ✅ K8s/Docker | ✅ Полное | Активен |
| **LangSmith** | SaaS | ❌ | ❌ Не рекомендуется | Cloud-only, EU/US |

### Обновления (2026)

- **Arize Phoenix:** Улучшена интеграция с OpenInference, добавлены enterprise-функции
- **Langfuse:** Вышла версия v3 (v2 EOL Q1 2025), улучшена производительность
- **Helicone:** Добавлен AI Gateway (open-source), улучшен кэшинг
- **SigNoz:** Полная поддержка GenAI-конвенций, улучшен UI

**Вывод:** Рекомендации по self-hosted инструментам корректны. LangSmith действительно не подходит для российского рынка из-за отсутствия опций локализации данных.

## 6. Дополнительные области валидации

### 6.1 Практики минимизации данных

**Подтверждено:**
- OpenTelemetry GenAI explicitly рекомендует **не записывать** системные инструкции, полные сообщения и вывод модели в атрибуты спанов по умолчанию
- Паттерн "внешнего хранилища контента" с ссылками в телеметрии — рекомендуемая практика
- Минимизация контента в логах и трассах снижает риски LLM02 (Sensitive Information Disclosure)

### 6.2 Инцидент-менеджмент и мониторинг

**Требуется уточнение в документах:**
- Runbook разбора инцидентов по `trace_id` — рекомендуется добавить
- Политика сэмплинга (полные трассы для ошибок / канареек) — рекомендуется добавить
- Ретенция журналов — требует согласования с ДПО

### 6.3 Связь с NIST AI RMF

**Подтверждено:**
- NIST.AI.600-1 (Generative AI Profile) — опубликован
- Сопоставление NIST AI RMF с ISO/IEC 42001 — доступно
- Функция Measure (измерение и мониторинг) — применима к GenAI

**Вывод:** Рекомендация не подменять 152-FZ и отраслевые требования РФ на NIST AI RMF — корректна.
