# Анализ Zero-Day уязвимости в LiteLLM и Telnyx: Атаки на ИИ-инфраструктуру через Supply Chain (Март 2026)

**Дата исследования:** 30 марта 2026 г.  
**Курс USD → RUB:** 85 RUB/USD  
**Назначение:** Технический и бизнес-анализ цепочки поставок атаки на популярные ИИ-пакеты LiteLLM и Telnyx с целью выявления уроков для обеспечения безопасности ИИ-инфраструктуры

## Executive Summary

В марте 2026 года threat actor группировка TeamPCP осуществила координированную атаку supply chain на два критически важных компонента ИИ-инфраструктуры: LiteLLM (популярный шлюз для маршрутизации запросов к LLM) и Telnyx (SDK для телефонии и AI Voice Agent). Атака продемонстрировала новую технику обхода традиционных средств защиты через стeganографию в аудиофайлах и привела к компрометации более 500 000 наборов учетных данных. Инцидент подчеркнул критическую важность поведенческого анализа кода и семантического сканирования зависимостей для защиты от zero-day атак в ИИ-экосистеме.

## 1. Технический разбор атаки

### 1.1 Хронология compromise

Атака TeamPCP следовала established pattern, начавшись с компрометации Trivy 19 марта 2026 года и последовательно распространяясь через npm, GitHub Actions и Checkmarx перед достижением PyPI:

- **19 марта**: Компрометация Trivy (уязвимость v0.69.4) и кража GitHub токенов
- **20-22 марта**: Распространение через self-propagating npm worm (CanisterWorm) и компрометация внутренних систем Aqua Security
- **23 марта**: Достижение Checkmarx и OpenVSX через GitHub Actions
- **24 марта**: Компрометация LiteLLM на PyPI (версии 1.82.7 и 1.82.8)
- **27 марта**: Компрометация Telnyx на PyPI (версии 4.87.1 и 4.87.2)

### 1.2 Техника атаки на LiteLLM

LiteLLM версии 1.82.7 и 1.82.8 содержали одинаковый вредоносный payload с разными механизмами активации:

**Вариант 1.82.7:**
- Вредоносный код внедрен непосредственно в `litellm/proxy/proxy_server.py`
- Активируется при использовании proxy модуля приложением
- Выполняет сбор credentials, локальное шифрование (AES-256 + RSA-4096) и exfilтрацию на `models.litellm[.]cloud`

**Вариант 1.82.8 (более опасный):**
- Вредоносный код размещен в файле `litellm_init.pth`
- Использует механизм Python sitecustomize для автоматического выполнения при старте интерпретатора
- Активируется сразу при импорте пакета, независимо от использования функционала
- Позволяет компрометировать системы даже без прямого вызова LiteLLM функций

### 1.3 Инновационная техника атаки на Telnyx

Для Telnyx версии 4.87.1 и 4.87.2 attackers применили новую технику обхода обнаружения:

- Вредоносный payload скрыт внутри математически валидных WAV аудиофреймов формата
- При импорте пакета выполняется запрос к `http://83.142.209.203:8080/ringtone.wav`
- Из аудиофайла извлекается второй-stage payload через XOR-расшифровку:
  ```python
  # Извлечение зашифрованных данных из WAV-фреймов
  with wave.open(wf, 'rb') as w:
      raw = base64.b64decode(w.readframes(w.getnframes()))
      s, data = raw[:8], raw[8:]  # Первые 8 байт - ключ, остальные - данные
      payload = bytes([data[i] ^ s[i % len(s)] for i in range(len(data))])
  ```
- Второй-stage payload harvests широкий спектр чувствительных данных:
  - SSH ключи и конфигурации
  - Облачные креденшлсы (AWS, GCP, Azure)
  - Authentication data от developer tools (Docker, npm, Git, Vault)
  - Database credentials
  - Environment configuration files (для извлечения embedded secrets)
  - Shell и database histories
  - Cryptocurrency wallet данные
  - Kubernetes service account tokens (приводящие к полной компрометации кластера)

### 1.4 Механизмы устойчивости и распространения

Обе атаки включали механизмы persistence и возможности для lateral movement:

**LiteLLM persistence:**
- Запись в `~/.config/sysmon/sysmon.py`
- Создание user systemd unit `sysmon.service`
- Beacon для follow-on payloads через polling `https://checkmarx[.]zone/raw`
- Способность создавать privileged `node-setup-*` pods в Kubernetes при наличии service-account токена

**Telnyx persistence:**
- Аналогичные механизмы через загруженный второй-stage payload
- Деплой привилегированных pod'ов в kube-system namespace при обнаружении Kubernetes токена
- Mounting host root filesystem и установка persistence implants непосредственно на узлы

## 2. Оценка воздействия и масштабы компрометации

### 2.1 Количественные показатели влияния

Согласно данным отчетов безопасности:
- LiteLLM пакет имеет около 97 миллионов ежемесячных загрузок на PyPI
- Telnyx широко используется в телефонии и AI Voice Agent приложениях
- Подтверждена компрометация более 500 000 наборов учетных данных
- Атака затронула разработчиков, CI/CD системы и production среды
- Особую опасность представляла версия LiteLLM 1.82.8 из-за автоматического выполнения при старте Python интерпретатора

### 2.2 Векторы эксплойта и поверхность атаки

Атака демонстрировала несколько критически важных векторов компрометации:

1. **Прямая кража учетных данных:** SSH ключи, облачные креденшлсы, API токены
2. **Компреметация CI/CD систем:** Через украденные publishing токены возможно дальнейшее распространение вредоносного кода
3. **Кластерная компрометация:** Через Kubernetes service account токены возможен деплой привилегированных pod'ов на всех узлах кластера
4. **Persistence mechanisms:** Системные сервисы и модифицированные конфигурационные файлы обеспечивают долгосрочный доступ
5. **Follow-on компрометация:** Beacon механизмы позволяют attackers загружать дополнительные payloads после первоначальной компрометации

## 3. Ответные меры и рекомендации по защите

### 3.1 Немедленные действия при компрометации

Организации, обнаружившие факты использования компрометированных версий, должны:

1. **Определить scope поражения:** Identify все хосты, контейнеры, CI jobs и developer рабочие станции, где установлены:
   - `litellm` версии 1.82.7 или 1.82.8
   - `telnyx` версии 4.87.1 или 4.87.2

2. **Оценить blast radius:** Проанализировать, какие credentials и ресурсы могли быть доступны скомпрометированным процессам

3. **Выполнить ротацию креденшлсов:** Отозвать и перевыдать любые учетные данные, достижимые от затронутого runtime, начиная с těch, которые могут:
   - Публиковать код
   - Доступ к CI/CD системам
   - Читать общие секреты
   - Создавать инфраструктуру

4. **Охота за Indicators of Compromise (IOCs):**
   - Network traffic to: `models.litellm[.]cloud`, `83.142.209[.]203`, `checkmarx[.]zone`
   - Filesystem artifacts: `litellm_init.pth`, `~/.config/sysmon/sysmon.py`, `/tmp/pglog`
   - Kubernetes activity: privileged pod creation с именами `node-setup-*`, доступ к secrets

### 3.2 Стратегические рекомендации для обеспечения безопасности ИИ-инфраструктуры

Based on the analysis of this supply chain attack, organizations should implement the following protective measures:

#### 3.2.1 Архитектурные меры защиты

**Строгая сегментация сети (Egress Filtering):**
- Ограничить исходящий трафик ИИ-gateway только авторизованными эндпоинтами провайдеров (OpenAI, Anthropic и т.д.) и внутренними сервисами
- Настроить строгие правила VPC и сетевые политики
- Даже если вредоносный пакет выполнит payload, он не должен иметь возможности связаться с внешними C2 серверами

**Изоляция сред выполнения:**
- Использовать sandboxed окружения для выполнения непроверенного кода
- Разделять среды разработки, тестирования и production с минимальными привилегиями
- Применять принцип наименьших привилегий к сервисным аккаунтам и пользовательским токенам

#### 3.2.2 Управление зависимостями и 공급 chain безопасность

**Жесткое управление версиями и хеш-проверка:**
- Фиксировать точные версии всех зависимостей в requirements.txt / poetry.lock
- Обязательно проверять хеши пакетов перед установкой
- Использовать инструменты вроде `pip hash` или поэзи-лока для верификации integrity

**Аудит транзитивных зависимостей:**
- Регулярно сканировать не только прямые зависимости, но и их зависимости
- Обращать особое внимание на пакеты с высоким уровнем доступа (встроенные в процесс выполнения, с правами на изменение системных файлов)
- Мониторить изменения в зависимостях через зависимости lock-файлов

#### 3.2.3 Внедрение поведенческого и семантического анализа

Переход от сигнатурного анализа к поведенческому:

**Инструменты семантического анализа кода:**
- Внедрять решения типа wtmp (who-touched-my-packages) от Point Wild
- Использовать LLM-powered анализ для выявления подозрительного поведения в коде зависимостей
- Задавать контекстные вопросы исходному коду: "Почему SDK для телефонии запускает XOR-расшифровку аудиофайла и передает результат в shell?"

**Интеграция в CI/CD пайплайны:**
- Добавлять семантический сканинг зависимостей как обязательный этап перед деплоем
- Использовать инструменты вроде Supply-Chain Firewall (SCFW) от Datadog для прозрачной блокировки известных вредоносных пакетов
- Внедрять continuous monitoring установленных пакетов в production средах

#### 3.2.4 Мониторинг и обнаружение угроз

**Расширенное журналирование и оповещение:**
- Мониторинг сетевого трафика на подозрительные домены и IP адреса
- Отслеживание файловых систем на появление известных артефактов вредоносного ПО
- Kubernetes аудит логов на привилегированные операции и несанкционированный доступ к секретам

**Поведенческая аналитика:**
- Аномалийное обнаружение в паттернах доступа к учетным данным
- Отслеживание нетипичных сеансов и геологингов
- Мониторинг процессов на нетипичное поведение (например, попытки чтения SSH ключей или облачных креденшлсов от неожиданных процессов)

## 4. Бизнес-влияние и рекомендации для принятия решений

### 4.1 Влияние на стратегию внедрения ИИ-решений

Инцидент с LiteLLM и Telnyx имеет несколько важных последствий для бизнес-стратегии:

**Риски доверяющих сторонних компонентов:**
- Даже популярные, widely-used пакеты могут быть скомпрометированы
- Традиционные меры доверия (popularity, количество загрузок) недостаточны для оценки безопасности
- Необходимо предполагать, что любой внешний компонент может быть скомпрометирован

**Влияние на выбор архитектуры:**
- Усиление аргументов в пользу on-premise и self-hosted решений там, где это возможно с точки зрения соответствия требованиям
- Оценка гибридных подходов с критической функциональностью, выполненной в изолированной среде
- Рассмотрение использования верифицированных, подписанных пакетов из trusted источников

**Изменение в подходах к управлению рисками:**
- Переход от реактивного к превентивному управлению supply chain рисками
- Внедрение continuous security assessment для всех внешних зависимостей
- Регулярное проведение red team упражнений, ориентированных на supply chain векторы атаки

### 4.2 Конкретные рекомендации для Comindware при работе с клиентами

Для исполнения миссии по внедрению и сопровождению ИИ-проектов заказчика Comindware должен:

1. **Обновлять методологию внедрения:**
   - Включать проверку поведенческой безопасности зависимостей в standard due diligence процесс
   - Добавлять семантический анализ кода в checklist безопасности перед внедрением ИИ-решений
   - Разрабатывать playbooks для реагирования на supply chain инциденты у клиентов

2. **Улучшать отчуждение ИИ (knowledge transfer):**
   - Обучать клиентские команды методам мониторинга и обнаружения supply chain угроз
   - Передавать знания о безопасных практиках управления зависимостями
   - Включать рекомендации по архитектурной изоляции в комплект отчуждения

3. **Развивать внутренние компетенции:**
   - Инвестировать в инструменты семантического анализа и поведенческого мониторинга
   - Развивать экспертизу в области AI безопасности и защиты от supply chain атак
   - Создавать внутренние базы знаний о последних угрозах и методах защиты

4. **Корректировать коммерческие предложения:**
   - Включать supply chain безопасность как стандартный компонент в AI реализации
   - Предлагать дополнительные услуги по мониторингу и защите ИИ-инфраструктуры
   - Различать предложения в зависимости от уровня требований к безопасности клиента

## 3.2.5 Решения для обнаружения zero-day атак в supply chain

Инцидент с LiteLLM и Telnyx продемонстрировал эффективность новых подходов к обнаружению вредоносного кода в зависимостях:

**Инструмент wtmp (who-touched-my-packages) от Point Wild:**
- Open-source CLI-утилита, выпущенная в ответ на атаку TeamPCP
- Меняет парадигму с поиска сигнатур на поведенческий анализ с помощью LLM (использует архитектуру LangGraph)
- Вместо сверки с черными списками CVE, система строит граф зависимостей Node/Python и задает контекстные вопросы самому исходному коду
- Примеры вопросов: «Почему SDK для телефонии запускает XOR-расшифровку аудиофайла и передает результат в shell?» или «Почему LLM-прокси читает SSH-ключи и переменные окружения?»
- Подход на базе LLM не является детерминированным на 100% и дает false-positives, однако в условиях zero-day атак он работает как "фонарик", подсвечивая скрытое вредоносное намерение (intent) кода, которое классические линтеры не видят в принципе
- Доступен на GitHub: https://point-wild.github.io/who-touched-my-packages/

**Дополнительные инструменты и подходы:**
- Supply-Chain Firewall (SCFW) от Datadog для прозрачной блокировки известных вредоносных пакетов
- Инструменты SBOM (Software Bill of Materials) для полной видимости зависимостей
- Интеграция семантического анализа в CI/CD пайплайны как обязательный этап предварительной проверки
- Continuous monitoring установленных пакетов в production средах на предмет аномального поведения
