# Быстрый старт: создание ассистента с доступом к знаниям и интеграцией с Comindware Platform {: #quick_start_rag_agent }

## Резюме {: #quick_start_summary }

- **Ситуация:** заказчику требуется переход от простых чат-ботов к интеллектуальным ассистентам, глубоко интегрированным с **Comindware Platform** и способным работать с любыми источниками знаний — от внутренних регламентов до внешних реестров (ФНС, 1С, Консультант+).
- **Вызов:** RAG в широком смысле — это не только поиск по базам знаний, а интеграция с любыми корпусами данных. Отсутствие чёткой методологии внедрения RAG-ассистентов и понимания затрат тормозит масштабирование.
- **Задача:** быстро собрать и развернуть промышленного ассистента, интегрированного с платформой **Comindware**, обеспечив при этом наблюдаемость, безопасность и экономическую эффективность.
- **Решение:** использовать модульную архитектуру на базе референс-стека **Comindware**. Начать с подготовки данных и бизнес-анализа, затем последовательно внедрить компоненты поиска и извлечения данных, инференса и интеграции с платформой.

## Необходимые компоненты {: #quick_start_components }

Для создания ассистента с доступом к знаниям и интеграцией с **Comindware Platform** требуются три слоя.

- **Слой данных:** подготовка, очистка и обогащение корпусов знаний (внутренние регламенты, Wiki, Jira, внешние реестры). Включает ETL-конвейеры, инкрементальную индексацию и векторизацию данных. Получение данных из внешних источников (ФНС, Консультант+, 1С) через API-коннекторы и преобразование в формат, пригодный для агента.
- **Слой агентской логики:** агенты, работающие с **Comindware Platform** через API и MCP. Агентный слой взаимодействует с данными и моделями платформы программно — читает и записывает сущности, создаёт записи, формирует резолюции, строит модели данных. Компоненты **агентного слоя Comindware** можно использовать как самостоятельные навыки (skills) для агентов общего назначения (например, OpenCode) или подключать по протоколу MCP — это позволяет интегрировать агентный слой в любой ИИ-ассистент, поддерживающий вызов внешних инструментов. Этот слой отвечает за асинхронную интеграцию с **Comindware Platform** (когда агент самостоятельно возвращает результат в платформу без каких-либо действий с её стороны).
- **Слой интеллектуальной обработки:** гибридный поиск (векторный + реранкинг), работа с документами в файловых хранилищах, промышленный инференс, наблюдаемость и безопасность. Также включает глубокий веб-поиск (deep research) для работы с источниками в интернете и генерацию кода в песочнице для обработки данных (в том числе автогенерацию SQL-запросов к источникам).

### Инвестиции в разработку (трудозатраты) {: #quick_start_r_d_investment }

Фактически затраченные ресурсы на создание ядра стека:

| Компонент | Затрачено времени (чел.-час.) |
| :--- | :--- |
| **Агентный слой Comindware Platform** | **320+** |
| **RAG-движок** | **900+** |
| **Итого инвестировано в стек** | **1220+** |

При внедрении у заказчика эти инвестиции не оплачиваются повторно: заказчик оплачивает только адаптацию готового стека под конкретные сценарии и бизнес-процессы (см. _«[Ресурсы и трудозатраты на сценарий](#quick_start_scenario_resources)»_).

!!! note "Возможности агентного слоя Comindware"

    Агентный слой умеет читать почти все сущности платформы, включая формы и схемы, но не модифицирует и не строит визуальные компоненты (формы, таблицы, BPMN-схемы) — он работает с данными и моделями данных, редактирует записи, формирует резолюции и выполняет глубокий веб-поиск. Реализовано около 50 инструментов из необходимых 100 для полного покрытия сущностей платформы.

## Архитектура решения {: #quick_start_architecture }

Референс-стек **Comindware** включает:

- **RAG-движок:** ядро системы, обеспечивающее многошаговое извлечение контекста, приоритизацию данных и стабильную индексацию с защитой от дублей.
- **Серверы инференса:** высокопроизводительный слой для обслуживания основных языковых моделей, специализированных вспомогательных моделей (эмбеддеры, ранжировщики результатов, классификаторы ПДн, анонимизаторы), охранных моделей (фильтрующих вредоносные запросы, спам и попытки джейлбрейка).
- **Наблюдаемость:** система трассировки запросов, оценки качества ответов и поиска «провалов» в базе знаний, должна включать следующие компоненты:
    - **Arize Phoenix** — слой трассировки, дашбордов и оценки качества LLM/RAG: обеспечивает детализированную трассировку по всей длине агентского цикла, дашборды качества RAG, эксперименты и оценки — всё из коробки, без написания самописного кода мониторинга.
    - **HiveTrace** — компонент защиты: обнаруживает промпт-инъекции, блокирует попытки джейлбрейка и утечки данных в реальном времени.

    **Phoenix и HiveTrace дополняют друг друга:** Phoenix отвечает за качество и отладку, HiveTrace — за безопасность агентной системы. Оба слоя трассируют токены, цены и метаданные и могут передавать данные как в платформу, так и во внешние системы наблюдаемости.

    Подробную стратегию промышленной наблюдаемости см. в _Приложении D «[Безопасность, комплаенс, наблюдаемость](./20260325-research-appendix-d-security-observability-ru.md#app_d_llm_rag_agent_observability)»_.

- **Безопасность:** контур модерации на базе охранных моделей. Классифицирует запросы и ответы на предмет вредоносных запросов, спама, попыток джейлбрейка, утечки данных и соответствия политикам компании.

!!! note "Текущее состояние наработок Comindware — PoC/MVP"

    Компоненты развёртываются в среде разработки без продуктовых артефактов и выпуска версий продуктов.
    
    Подготовка контейнеров Docker и манифестов Kubernetes для промышленной эксплуатации — в планах.

## RAG — универсальный ETL-конвейер данных {: #quick_start_etl }

Бизнес-данные — главный источник добавленной стоимости ИИ-решений.

Агент без доступа к актуальным данным превращается в чат-пустышку.

Качество ответов агента на 80% зависит не от выбора модели, а от качества подготовки и обработки данных.

Поэтому этап ETL (Extract, Transform, Load — извлечение, преобразование, загрузка) и интеграции с внешними системами занимает львиную долью бюджета и трудозатрат проекта и определяет его успех.

!!! warning "Стройте архитектуру ETL в первую очередь"

    Этап ETL необходимо начинать задолго до создания любых ИИ-агентов, фокусируясь на глубоком бизнес-анализе состава, характера и потоков данных.

    Только после формирования архитектуры данных и правил их извлечения, очистки, обогащения и обновления (инкрементальные конвейеры) можно переходить к технической реализации.

### Анализ и планирование конвейеров данных {: #quick_start_etl_planning }

Прежде чем приступать к индексации или извлечению данных, команда аналитиков и инженеров данных проводит следующие работы:

- **Инвентаризация датасетов:** определение корпоративных наборов данных (Wiki, регламенты, прошлые решения поддержки, базы 1С) и их предварительная оценка — валидация достоверности и разметка критериев «хорошо/плохо» для каждого конкретного кейса и бизнеса заказчика. Эти наборы данных становятся основой для системных и промежуточных промптов и контекстного инжиниринга.
- **Инвентаризация динамических источников:** помимо статичных документов, RAG-контур может получать динамические данные через API, например:
    - **ФНС (ЕГРЮЛ/ИП, РНП):** проверка статуса контрагентов, получения выписок и мониторинга изменений в реестре.
    - **1С (Предприятие 8.3+):** выгрузка данных для интеграции с складскими остатками, заказами и справочниками контрагентов..
    - **Консультант+/Гарант:** получение из справочно-правовых систем получения актуальных редакций документов и судебной практики.
- **Семантический аудит:** анализ терминологии, синонимов и специфического сленга компании.
- **Проектирование трансформаций:** разработка правил очистки (удаление устаревших версий, очистка от мета-мусора) и обогащения данных (автоматическое добавление тегов категорий, связей с продуктами).
- **Выбор комбинации подходов к индексации:** файловый поиск (агент ищет документы средствами файловой системы), гибридный поиск и индексация (по векторам и  ключевым словам), API (данные извлекаются из внешних систем по запросу).
    - Файловый поиск подходит для часто меняющихся или добавляющихся корпусов знаний в виде файлов.
    - Гибридный поиск подходит для корпусов где новые знания преимущественно добавляются, а старые знания остаются стабильными.
    - Извлечение по API подходит для динамических источников (ФНС, 1С, Консультант+).

## Интеграция с **Comindware Platform** {: #quick_start_platform_integration }

Ключевая особенность нашего решения — **асинхронное взаимодействие** агента с **Comindware Platform**.

- **YAML-схемы:** вся логика взаимодействия и сопоставление сущностей агента и **Comindware Platform** представлены в декларативных конфигурационных файлах. Это позволяет менять структуру полей в приложениях без переписывания кода агента.
- **Асинхронный вызов:** **Comindware Platform** инициирует задачу и продолжает работу. Агент в фоновом режиме собирает данные, рассуждает, формирует и проверяет выводы, а затем сам записывает результат обратно в **Comindware Platform**, создавая связанные записи и выставляя любые требуемые статусы.

При построении архитектуры важно **определить границы:** что выполняет сама **Comindware Platform** по своим сценариям и схемам, а что делегируется агенту.

- **Comindware Platform** обрабатывает рутинные операции и предсказуемые маршруты внутри себя (валидация данных, маршрутизация заявок, вычисления по формулах), снижая вычислительную нагрузку и трафик между агентом, внешними источниками и платформой.
- Агент вступает в работу там, где требуется рассуждение, недетерминированный поиск и анализ данных, сложные непредсказуемые цепочки обращений к внешним источникам.

## Этапы реализации {: #quick_start_phases }

Организационные рамки и фазы внедрения соответствуют _[Методологии разработки и внедрения ИИ](./20260325-research-report-methodology-main-ru.md#method_pack_overview)_.

Ниже перечислены ключевые этапы от идеи до работающего в продакшене агента:

!!! tip "Критерии приёмки и артефакты"

    Каждый этап имеет критерии приемки и артефакты.
    
    Не переходите к следующему этапу, пока не зафиксированы результаты текущего — это снижает риск переделок и перерасхода бюджета.

- **Подготовка данных и ETL:** фундамент проекта. Сбор, очистка и обогащение данных из всех источников (внутренние регламенты, Wiki, Jira, внешние реестры). На этом этапе определяется качество будущих ответов агента — без качественных данных даже самая мощная модель будет давать нерелевантные результаты.
- **Архитектура и проектирование:** определение «границ агентности» — что агент делает сам, а где требуется эскалация к человеку, а также что выполняет сама платформа по своим сценариям и схемам, а что делегируется агенту. Балансировка вычислительной нагрузки и трафика между платформой, агентом и внешними источниками позволяет избежать избыточных вызовов модели. Проектирование схемы данных в платформе и правил взаимодействия агента с внешними источниками (API ФНС, 1С, справочные системы).
- **Настройка поиска и реранкинга:** настройка стратегий извлечения контекста и моделей реранкинга, чтобы из сотни найденных фрагментов выбрать 5–10 наиболее релевантных. Это повышает точность ответов на 15–20&nbsp;% и снижает потребление токенов.
- **Безопасность и наблюдаемость:** настройка контуров модерации (фильтрация вредоносных запросов, спама, попыток джейлбрейка, утечек ПДн) и системы наблюдаемости (трассировка запросов, оценка качества ответов). Эти компоненты закладываются с первого дня пилота, а не добавляются после завершения.
- **Инференс и масштабирование:** выбор модели размещения (локально, облако, гибрид) и оптимизация серверов инференса. Применение квантования позволяет запускать крупные модели на стандартном оборудовании без потери качества.
- **Интеграция с платформой и оркестрация:** соединение всех компонентов в единый конвейер и настройка маппинга полей ассистента в атрибуты **Comindware Platform**. Агент работает асинхронно: платформа инициирует задачу, агент выполняет работу в фоне и записывает результат обратно.
- **Передача знаний и запуск:** передача команде заказчика эксплуатационных регламентов — как обновлять индекс, как читать трассы наблюдаемости, как менять промпты и политики безопасности.

## Практический сценарий: ассистент поддержки с проверкой контрагентов {: #quick_start_scenario }

### Описание кейса {: #quick_start_scenario_description }

**Бизнес-задача:** автоматизировать обработку заявок на закупку, требующих проверки благонадежности поставщика по открытым источникам.

**Алгоритм работы агента:**
1.  **Входящий триггер:** в **Comindware Platform** создается запись «Заявка на закупку» с ИНН поставщика.
2.  **Асинхронный запуск:** платформа дергает агента. Агент забирает ИНН и название компании.
3.  **Поиск в БЗ:** агент ищет в базе знаний регламенты закупки для данной категории товаров.
4.  **Внешняя проверка (вызов инструмента):** агент вызывает инструмент «ФНС-Проверка», передает ИНН и получает актуальный статус (действующая, в стадии ликвидации, наличие в реестре недобросовестных поставщиков).
5.  **Углублённый веб-поиск:** если статус компании спорный, агент запускает поиск в интернете (новости, отзывы, арбитражные дела).
6.  **Формирование вывода:** агент собирает все факты: «Регламент допускает закупку, ФНС подтверждает статус 'Действующая', арбитражных дел нет. Рекомендовано: Одобрить».
7.  **Запись результата:** агент пишет ответ в поле «Резолюция ИИ» и выставляет статус «Проверено: Безопасно».

### Ресурсы и трудозатраты {: #quick_start_scenario_resources }

Для реализации подобного сценария «с нуля» до рабочего прототипа на базе готового стека **Comindware** требуются следующие ресурсы:

- **Бизнес-аналитик (1 чел.):** сбор требований, подготовка эталонного набора тестовых вопросов, разметка данных. **Трудозатраты:** 40–60 чел.-часов.
- **ИИ-инженер / Разработчик (1 чел.):** настройка ETL, тюнинг промптов, интеграция с API ФНС, отладка RAG-конвейера. **Трудозатраты:** 80–120 чел.-часов.
- **DevOps / Системный администратор (0,5 чел.):** развертывание серверов инференса, систем наблюдаемости и безопасности, настройка GPU-окружения. **Трудозатраты:** 20–30 чел.-часов.

**Итоговый срок реализации:** 3–4 недели до запуска пилотной версии.

## Состав и роли проектной команды {: #quick_start_team_roles }

Для успешного внедрения и эксплуатации корпоративного ИИ-агента рекомендуется формирование компактной кросс-функциональной команды со следующими зонами ответственности:

| Роль | Ключевые компетенции | Зона ответственности в проекте |
| :--- | :--- | :--- |
| **Аналитик-промптолог (ИИ/RAG)** | Бизнес-анализ, промпт-инжиниринг, работа с данными | Описание бизнес-логики, подготовка эталонного набора вопросов, оценка качества ответов, разметка данных для RAG |
| **Инженер-разработчик (LLM/Python)** | Python, LangChain/LlamaIndex, SQL, API-интеграции | Настройка ETL-конвейеров, реализация инструментов (Tools), интеграция с **Comindware Platform**, тюнинг стратегий поиска |
| **Инженер инфраструктуры (DevOps/LLMOps)** | Docker, Kubernetes, мониторинг | Развертывание и масштабирование серверов инференса, управление GPU-ресурсами, настройка наблюдаемости |
| **Эксперт предметной области** | Глубокое знание предметной области (напр., юрист или инженер ТП) | Верификация экспертных знаний в базе, финальная приёмка ответов на соответствие регламентам |
| **Руководитель проекта** | Управление ИТ-проектами, гибкие методологии (Agile) | Координация фаз (концепт → пилот → масштабирование), управление ожиданиями стейкхолдеров, контроль соблюдения SLA/SLO |
| **Инженер по качеству (QA/Eval)** | Тестирование ПО, метрики качества LLM (RAGAS, DeepEval) | Регрессионное тестирование, автоматизированная проверка на «галлюцинации», контроль дрейфа качества |

**Оптимальный состав для старта:** 3–4 специалиста (аналитик, разработчик и инженер могут совмещать роли на этапе концепта). Ключевой фигурой является **аналитик-промптолог**, так как именно качество данных и промптов определяет итоговую ценность для бизнеса.

### Риски внедрения {: #quick_start_risks }

1. **Технический риск (деградация качества):** со временем база знаний устаревает, а модели могут начать «галлюцинировать». **Решение:** постоянный мониторинг через наблюдаемость и еженедельное регрессионное тестирование на эталонном наборе.
2. **Операционный риск (фактор ключевого сотрудника):** зависимость проекта от одного инженера. **Решение:** использование стандартизированного стека **Comindware** и ведение журнала архитектурных решений.
3. **Инфраструктурный риск:** дефицит или поломка GPU-мощностей. **Решение:** настройка гибридного инференса с облачным резервированием.
4. **Риск дрейфа данных:** изменение формата входящих данных (например, новые типы заявок), к которым агент не готов. **Решение:** отслеживание ошибок обработки входящих данных с автоматизированными оповещениями; отслеживание аномального снижения оценки достоверности ответов.
