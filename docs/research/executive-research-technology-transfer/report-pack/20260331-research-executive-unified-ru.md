# Резюме для руководства: коммерческое обоснование, методология и экономика внедрения ИИ {: #exec_unified }

Документ даёт руководству **Comindware** единые коммерческие рамки для продажи и передачи экспертизы по корпоративному ИИ: что именно предлагается клиенту, по каким этапам внедряется контур, как обосновывается экономика (CapEx/OpEx/TCO) и по каким критериям фиксируется передача владения и операционной способности заказчику.

Цель документа — поддержать управленческие решения в сделке и запуске внедрения, а не заменить детальные технические и сметные разделы основных отчётов.

## Решение для руководства за 60 секунд {: #exec_unified_decision_card }

- **Коммерческое решение:** **Comindware** продаёт не «доступ к модели», а промышленное внедрение с передачей способности заказчику эксплуатировать и развивать контур (KT/IP/BOT).
- **Операционная траектория:** PoC -> Пилот -> Масштабирование -> BOT/Create-Transfer; принимайте решение о переходе между этапами по метрикам качества, утилизации и экономики.
- **Финансовая дисциплина:** используйте CapEx/OpEx/TCO-вилки как порядок величин; фиксируйте договорные значения только после стендовых замеров и проверки прайсов на дату.
- **Ключевая оговорка:** используйте глобальные enterprise-бенчмарки только как контекст рынка, не как договорную норму для резидентного контура РФ.
- **Следующее управленческое решение:** утвердить 30/60/90-план, владельцев TOM и критерии приёмки передачи.

!!! warning "Единая граница применимости"

    В РФ приоритетны 152-ФЗ, резидентность данных и локальные тарифы; зарубежные и глобальные ориентиры используются только как сравнительный бенчмарк.

## SCQA для коммерческого решения {: #exec_unified_scqa }

- **Ситуация:** спрос на GenAI растёт, но у заказчиков дефицит управляемого внедрения и масштабирования.
- **Проблема:** без прозрачной экономики, контроля качества и формализованной передачи пилоты не капитализируются в устойчивый внутренний актив.
- **Вопрос:** как внедрить корпоративный ИИ с предсказуемой экономикой и контролируемой передачей владения?
- **Ответ:** поэтапная программа внедрения с едиными KPI, формальными критериями приёмки и пакетом отчуждения.

## Бизнес-ценность **Comindware** {: #exec_unified_offer }

- **Внедрение:** от PoC к промышленному контуру с контрольными точками качества и наблюдаемости.
- **Эксплуатация и рост:** TOM, роли, FinOps, eval/регрессии, управление рисками.
- **Передача владения:** код, конфигурации, runbook, обучение, критерии приёмки, юридическая чистота контура.
- **Суверенный контур РФ:** архитектура и данные проектируются под требования резидентности и комплаенса.

## Пакеты по этапам внедрения {: #exec_unified_packages }

### Пакет 1. Управленческая диагностика + выбор 2-3 кейсов {: #exec_unified_package_1 }

**Результат:** согласованный набор приоритетных сценариев, KPI/ограничения, требования к данным и комплаенсу, контур принятия решений.

### Пакет 2. PoC (2-4 недели) {: #exec_unified_package_2 }

**Результат:** рабочий прототип на 1-2 кейсах с первичным измерением эффекта, стоимости и рисков.

### Пакет 3. Пилот (1-3 месяца) {: #exec_unified_package_3 }

**Результат:** пилот в среде, приближенной к продакшну: интеграции, наблюдаемость, первые пользователи, базовые метрики.

### Пакет 4. Масштабирование (3-12 месяцев) {: #exec_unified_package_4 }

**Результат:** промышленный контур с TOM (роли/процессы/KPI), модель сопровождения, контроль качества и план расширения.

### Пакет 5. BOT / Create-Transfer (передача и отчуждение) {: #exec_unified_package_5 }

**Результат:** формализованный комплект передачи и критерии приёмки, закрепляющие способность заказчика эксплуатировать и развивать контур самостоятельно.

## Матрица аргументов по ролям ЛПР заказчика {: #exec_unified_role_matrix }

| Роль | Что важно | Фокус аргумента | Аргумент из комплекта |
| :--- | :--- | :--- | :--- |
| **CEO** | P&L, капитализация, независимость | Го/нет-го на этапах внедрения | ИИ-контур становится внутренним активом, а не внешней подпиской. |
| **CFO** | Бюджет, TCO, владение активами | Границы CapEx/OpEx и пороги окупаемости | При стабильной нагрузке собственный/гибридный контур может стать выгоднее SaaS-потребления. |
| **CRO** | Упаковка и переговоры | Этапные пакеты + бюджетные вилки | Передача и суверенность повышают ценность сделки для клиента. |
| **CPO** | Roadmap, качество, ROI | Приоритизация сценариев и критерии масштаба | Масштабирование опирается на измеримый эффект, а не демо-результаты. |
| **COO** | Операционная устойчивость | Готовность процесса после передачи | Роли, эскалации и runbook обеспечивают устойчивую эксплуатацию. |
| **CIO / CTO** | Архитектура и управляемость | Облако РФ / on-prem / гибрид | Контроль данных, интеграций, observability и жизненного цикла знаний. |
| **CISO** | Периметр, комплаенс, безопасность | Политики телеметрии и данных | 152-ФЗ, минимизация данных, контроль обработки ПДн, OWASP/NIST-ориентиры. |

## Артефакты передачи (что останется у заказчика) {: #exec_unified_artifacts }

- Исходный код и манифест зависимостей (воспроизводимая сборка).
- Конфигурации без секретов + перечень env-переменных.
- Регламент эксплуатации (runbook): старт/стоп/backup/масштабирование/инциденты.
- Наборы eval/регрессии с базовым уровнем и критериями деградации.
- Политика наблюдаемости: сэмплинг, ретенция, маскирование ПДн, дашборды/алерты.
- Политика данных и индексации RAG: ingestion, обновления, права доступа.
- Роли/эскалации после передачи.
- Программа обучения: бизнес, эксплуатация, разработка, ИБ.

## Экономическое обоснование сделки (CapEx / OpEx / TCO) {: #exec_unified_economics }

**Финансовая база:**

- Бюджетное обоснование опирается на публичные тарифы облаков РФ, профили GPU и сценарный сайзинг.
- Разделяйте затраты на разовые и повторяющиеся: инфраструктура, интеграции, сопровождение, безопасность/eval, токены/API.

**Практический ориентир выбора модели владения:**

- Для входа в проект базовый сценарий — облачный PoC в РФ.
- Для устойчивой нагрузки при утилизации >60 % рассматривайте переход к гибриду/on-prem.
- Для решения CFO/CIO сопоставляйте сценарии по единым допущениям и валютной политике.

**Рамка применимости цифр:**

- Вилки и бенчмарки задают порядок величин для переговоров и бюджетного коридора.
- Фиксируйте финальные значения для КП после сверки прайсов на дату и стендовых замеров под целевой SLO.

## Числовые пороги решения (единый benchmark) {: #exec_unified_guardrails }

- **Утилизация:** целевой ориентир >60 % регулярного использования в целевой группе.
- **Эффективность:** целевой диапазон 30-40 % сокращения времени выполнения типового сценария.
- **Качество:** целевой ориентир >95 % по внутренней рубрике (LLM-as-judge) при стабильной методике оценки.
- **TCO break-even:** рассматривайте переход к on-prem/гибриду при устойчивой нагрузке и подтверждённой утилизации >60 %.

## Типовые возражения и ответы {: #exec_unified_objections }

**«Сложно обосновать ROI».**
Сначала используйте диапазоны CapEx/OpEx/TCO для раннего решения; точный ROI фиксируйте после замеров на стенде заказчика и базового цикла пилота.

**«Будет vendor lock-in».**
Передача по BOT/Create-Transfer включает код, конфигурации, runbook, eval и обучение; клиент получает операционную автономию в контуре.

**«Риски ИБ и 152-ФЗ слишком высокие».**
Архитектура строится под требования заказчика: периметр до LLM, минимизация данных, маскирование и политика журналирования; для управляемых API проверьте отдельно договорной контур обработки данных.

## Рыночные сигналы для переговоров (контекст, не норма КП) {: #exec_unified_market_limits }

- Российский спрос на GenAI растёт, но дефицит смещён в область внедрения и масштабирования под требования комплаенса и экономики.
- В инфраструктуре РФ усиливается предложение управляемых LLM-платформ и enterprise-инструментов (примерные ориентиры описаны в _«[Приложении E](./20260325-research-appendix-e-market-technical-signals-ru.md#app_e_root)»_ и отчёте _«[Методология разработки и внедрения ИИ](./20260325-research-report-methodology-main-ru.md#method_pack_overview)»_).
- Используйте международные отчёты (OpenAI, McKinsey, Stanford) как сравнительный контекст, а не как юридически значимое обещание результата в КП.

## Коммерческий план 30/60/90 {: #exec_unified_action_plan }

### 0-30 дней {: #exec_unified_action_30 }

- Проведите workshop и выберите 2-5 приоритетных сценариев.
- Аудируйте готовность данных и требования 152-ФЗ.
- Зафиксируйте baseline: время выполнения сценария, эскалации, текущую стоимость.
- Согласуйте KPI, состав пилота и критерии перехода между этапами.

### 30-60 дней {: #exec_unified_action_60 }

- Разверните PoC/пилотный контур (RAG + инференс + наблюдаемость) в выбранной среде.
- Проведите техническую и бизнес-валидацию сценариев.
- Подготовьте первичную оценку ROI и управленческое решение: расширение сделки или стоп до фазы пилота.

### 60-90 дней {: #exec_unified_action_90 }

- Запустите промышленный пилот на реальной нагрузке.
- Подтвердите целевые метрики качества/экономики/утилизации.
- Формализуйте комплект передачи, график отчуждения и модель сопровождения.
- Утвердите дорожную карту масштабирования в контуре заказчика.
