# Отчёт. Сайзинг и экономика (CapEx / OpEx / TCO)

## Описание документа {: #sizing_intro_and_summary }

### Роль документа в комплекте {: #sizing_pack_overview }

Документ даёт **количественные опоры** для решений по бюджету: тарифы облаков РФ, вилки CapEx/OpEx, дерево факторов стоимости и сценарии TCO для RAG, облачного и on-prem инференса, агентских контуров.

### Назначение {: #sizing_purpose_and_scope }

Документ обобщает **модель затрат и экономику эксплуатации** для корпоративных ИИ-контуров в резидентной логике РФ: облако, on-prem и гибрид. Представленные показатели — это **диапазоны для управленческих решений** и сценарного сравнения, а не фиксированная коммерческая смета или оферта.

**Практический смысл документа:** дать единую финансово-техническую базу для выбора архитектуры, согласования CapEx/OpEx с CFO/CIO/CTO и подготовки защищаемого TCO-обоснования по этапам внедрения. Финальное инвестиционное обязательство принимается после актуализации тарифов и подтверждения нагрузочного профиля.

### Как использовать для управленческих решений {: #sizing_how_to_use }

Используйте документ как канонический источник по тарифам, TCO, порогам утилизации и чувствительности к курсу. Для C-level переносите из него бюджетные вилки и правила сравнения сценариев, а не отдельные технические таблицы.

!!! warning "Ограничения"

    Не используйте отчёт как:

    - Точную смету без замеров на стенде заказчика (вилки — ориентир, не контрактная цифра).
    - Базовую смету для КП без актуализации прайса провайдера.

### Логика решения (SCQA) {: #sizing_scqa }

**SCQA (Situation–Complication–Question–Answer)** — ситуация → проблема → вопрос → ответ.

- **Ситуация:** TCO GenAI складывается из токенов (API), GPU (аренда или on-prem), интеграции и постоянного сопровождения; для стека **Comindware** профиль нагрузки задают **корпоративный RAG‑контур** и **агентный слой Comindware Platform**, слой инференса — **MOSEC** / **vLLM**; для РФ добавляются локальные прайсы и требования к данным.
- **Проблема:** стоимость **вспомогательных моделей** (эмбеддинг, реранг, гард) и **мульти‑бэкенд**‑размещение часто недооцениваются; «скрытые» статьи (безопасность, оценка качества, миграции моделей), **колебания курса** искажают сравнение облака и своего контура; без учёта **доли запросов** на «лёгком» контуре смешиваются сценарии с разным качеством и комплаенсом; глобальные бенчмарки — контекст рынка, **не** автоматическая норма для резидентного КП.
- **Вопрос для решения:** какой режим финансирования и какой сайзинг выбрать клиенту (облако РФ vs on‑prem vs гибрид), какие **диапазоны** CapEx/OpEx заложить на 1–3 года и как обосновать чувствительность.
- **Рекомендуемый ответ:**
    - начинать с облачного PoC по российским тарифам, затем при устойчивой высокой утилизации **порядка >60 %** и горизонте владения **несколько лет** оценивать безубыточность on‑prem;
    - опираться на дерево факторов стоимости, сценарный сайзинг и раздел TCO ниже, включая **строки наблюдаемости** (телеметрия, ретенция, сэмплирование) как часть OpEx и **строки пре‑LLM** (CPU/GPU для вспомогательных моделей, хранилище плейсхолдеров, регрессионные тесты при смене бэкенда);
    - опора на **сценарные диапазоны** (полосы неопределённости), единые определения CapEx и OpEx, матрицу решений с **согласованными вилками**;
    - глобальные цифры — с **короткой формулой области выборки**;
    - зарубежные API‑шлюзы (**OpenRouter**, **OpenCode Zen**) учитывать отдельно от тарифов облаков РФ — см. _«[Зарубежные API (разработка и песочница)](#sizing_foreign_api_dev_sandbox)»_;
    - измеримый бизнес‑эффект требует **организации зрелости** — см. отчёт _«[Методология разработки и внедрения ИИ](./20260325-research-report-methodology-main-ru.md#method_ai_strategy_org_maturity)»_; финальные цифры КП — после прайса провайдера на дату и, для on‑prem, замеров на целевом стеке.

### Управленческие компромиссы {: #sizing_management_tradeoffs }

- **SMB / департамент** — низкий CapEx, оплата по токену vs меньше контроля над данными и вендор-лок на API.
- **Enterprise** — инвестиция в GPU и команду LLMOps vs снижение долгосрочного TCO и суверенитет.
- **Гибрид** — гибкость vs сложность учёта и двойной обсервабильности.
- **Открытые веса (MIT) vs управляемый API** — без счётчика токенов за сам чекпойнт vs CapEx/аренда GPU и инженерия; вилка разобрана в разделе «**Открытые веса и API: влияние на TCO**» ниже ([первичное описание релиза — Хабр, Сбер](https://habr.com/ru/companies/sberbank/articles/1014146/)).

**Примеры метрик:** стоимость 1 млн токенов по выбранному провайдеру, утилизация GPU, полные 3-летние TCO on-prem vs cloud, чувствительность к курсу и пошлинам (раздел по РФ).

**Ключевой вывод:** для SMB оптимальны российские облака (ориентир от ~12 ₽/млн токенов на отдельных линейках, см. таблицы); для крупного Enterprise решение о переходе к on-prem или гибриду корректно принимать не по «быстрому сроку окупаемости», а по **устойчивой утилизации**, горизонту владения и **полному TCO** с учётом энергии, персонала и амортизации. Использование формата TOON вместо JSON может **заметно** снижать расход токенов в структурированных обменах, но фактический эффект зависит от формы данных и токенизатора модели (см. раздел «Сборник мер по оптимизации стоимости» ниже).

### Ролевой фокус ЛПР {: #sizing_c_role_decisions }

Полная матрица приоритетов ЛПР приведена в _[методологическом отчёте](./20260325-research-report-methodology-main-ru.md#method_c_role_decisions)_; ниже — финансовые акценты для решения по модели владения и бюджету.

- **CFO:** границы CapEx/OpEx, пороги утилизации, чувствительность к курсу и прайсу.
- **CIO/CTO:** стоимость архитектурных вариантов (cloud/on-prem/hybrid), эффект на SLA и операционную нагрузку.
- **CEO/CRO/CPO/CISO:** экономические последствия выбранной модели владения и ограничений комплаенса.

### Компоненты стоимости внедрения ИИ-агента {: #sizing_ai_agent_implementation_costs }

6.  **Эффективность:** Юнит-экономика одного запроса.

### Барьеры и эффекты внедрения (глобальные показатели) {: #sizing_adoption_barriers_effects_global }

По данным глобальных исследований (McKinsey, Stanford, EY, OECD), ключевые барьеры и эффекты:

- **Барьеры (качество и безопасность):** **40-50%** компаний отмечают проблемы качества и галлюцинаций как значимый барьер; **45-60%** выражают опасения по поводу утечки данных и безопасности (Stanford: +56,4% инцидентов год к году; Cisco: 60% обеспокоены уязвимостями).
- **Эффекты (производительность):** опросы EY и Deloitte фиксируют **восприятие** роста продуктивности в диапазоне **40-66%**, но такие оценки нельзя напрямую сопоставлять с макроуровнем. OECD моделирует **долгосрочный** вклад ИИ в annual aggregate labour productivity growth на горизонте **10 лет**: для наиболее AI-exposed экономик G7 диапазон составляет примерно **0,4-1,3 п.п. в год** ([OECD, 2025](https://www.oecd.org/en/publications/macroeconomic-productivity-gains-from-artificial-intelligence-in-g7-economies_a5319ab5-en.html)). Для пилотов и КП это означает: компании чаще видят локальное ускорение задач раньше, чем подтверждённый макроэффект.

## Экономический каркас {: #sizing_economic_framework }

Стоимость ИИ-решения складывается из трех уровней:

1.  **Инфраструктура (Infrastructure)**
    - GPU Compute (VRAM, Bandwidth)
    - Vector DB Storage (ChromaDB)
    - Network (Egress fees, Latency)
2.  **Инференс (Inference)**
    - Стоимость токенов (Input/Output)
    - Эмбеддинги и Реранкинг
    - Оверхед фреймворков (vLLM/MOSEC)
    - **Пре-LLM и вспомогательный NLP:** детерминированное маскирование (маржинальный CPU), при необходимости малые NER/классификаторы на CPU или GPU, **хранение карт соответствий** плейсхолдеров (память процесса, Redis и т.п., TTL сессий) — влияет на p95/p99 задержки до вызова основной LLM
    - **Мульти-бэкенд:** когда разные классы моделей обслуживаются разными процессами или движками, растут суммарный **оверход VRAM**, число релизных единиц и трудоёмкость регрессионного тестирования при обновлении стека
3.  **Эксплуатация (Operations)**
    - LLMOps / DevOps поддержка
    - Мониторинг и observability (например, Langfuse, Phoenix, LangSmith — по выбору и правовой оценке контура)
    - Хранение трасс и аудит-журналов агентских цепочек (политика ретенции, объём диска, запросы при расследованиях, маскирование персональных данных в журналах) — отдельный множитель OpEx при глубокой наблюдаемости
    - **Объём телеметрии × ретенция:** число запросов, средняя глубина спанов на агентский прогон, срок хранения и репликация индексов бэкенда трасс — прямой драйвер **СХД** и **лицензий/квот** SaaS; архитектуры **исследовательского цикла над долговременной памятью** (иллюстрация — [General Agentic Memory, GAM](https://arxiv.org/pdf/2511.18423)) увеличивают типичное число **раундов инструмент+LLM** на пользовательский запрос и тем самым **глубину спанов** и токены относительно «одного» RAG-ответа.
    - **Сэмплинг:** head-based (дешевле, теряет редкие сбои) vs tail-based / полные трассы для ошибок и канареек; влияет на полноту разборов и на счёт за хранение.
    - **Раздельные бэкенды** для dev/stage/prod и запрет экспорта содержимого промптов из прода в недоверенные системы — снижает риск нарушения 152-ФЗ, но дублирует фиксированный OpEx на стенды.
    - **Экспорт в SIEM / корпоративный мониторинг** и конвейеры **маскирования/DLP** для полей телеметрии — дополнительные вычисления и персонал ИБ.
    - Обновление и дообучение моделей
    - **Мультиагентные прогоны в обвязке разработки:** суммарные токены × число итераций, время на стене, нагрузка на **CI** и **браузерные исполнители** при **сквозных** проверках; публичные сравнения «один агент против полной обвязки» дают лишь **порядок** различия затрат и времени, не норму для коммерческого предложения ([Anthropic — Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)).
    - **Настройка модели-контролёра:** время специалистов на промпты, **эталонные примеры в промпте** и чек-листы по критериям — иначе растёт риск **завышенных** вердиктов и повторных циклов ([Anthropic — Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)).
    - **Регулярная синхронизация док ↔ код («садовник» / сборка мусора):** вычислительные джобы и (или) человеко-часы на поддержание актуальности знаний в репозитории ([OpenAI — Harness engineering](https://openai.com/ru-RU/index/harness-engineering/), [Хабр — обвязка для агентов](https://habr.com/ru/articles/1005032/)).
    - **Структурированные некодовые артефакты (пример: BPMN 2.0 XML):** ревью и доработка после генерации LLM, если в промпте не зафиксированы согласованность семантики и **BPMNDI** — закладывать как **инженерное время** в TCO (см. параграф _«[Формализация процессов (BPMN 2.0) и генерация с помощью LLM](./20260325-research-report-methodology-main-ru.md#method_bpmn_process_formalization_llm)»_ отчёта «Методология разработки и внедрения ИИ»).
    - **Сеть и удостоверение для песочниц агентов:** проектирование egress, интеграция с IAM, эксплуатационный регламент инцидентов и сопровождение политик — закладывать как **инженерное время** (OpEx), если в контуре заказчика есть **недоверенное исполнение** (см. _Приложении D, параграф «[Граница доверия, сеть и среда исполнения агента](./20260325-research-appendix-d-security-observability-ru.md#app_d__trust_boundary_agent_environment)»_).
    - **Платформа задач для изолированного исполнения агентов:** минимальный контур (политики, планировщик сред, реестр образов, брокер доступа, хранилище результатов, аудит) — закладывать как **CapEx / облачные сервисы** и **инженерное время** ввода; **долгоживущие** dev-среды со снимками и SSO добавляют **хранение**, **лицензии IAM** и сопровождение жизненного цикла (см. _Приложении D, параграф «[Модель риска, паттерны среды и минимальный состав платформы](./20260325-research-appendix-d-security-observability-ru.md#app_d__risk_model_platform_patterns)»_).
    - **Управляемые песочницы (SaaS):** тарификация по **времени жизни** среды, объёму **хранения** снимков и исходящему **трафику**; в юнит-экономике отдельно учитывать **p50** время создания/восстановления и **p95** хвоста, **долю успешных** прогонов по типам задач, **стоимость одного** успешного выполнения и риск **сбоев восстановления** из снимка (см. _Приложении D, параграф «[Управляемые песочницы, сравнение моделей и бенчмарки](./20260325-research-appendix-d-security-observability-ru.md#app_d__managed_sandboxes_benchmarks)»_).
    - **Предпрогретые среды (warm pool) vs контейнеры с более слабой изоляцией:** удержание **готовых** экземпляров снижает **холодный** старт и улучшает предсказуемость latency, но добавляет **постоянные** затраты на простой и усложняет гигиену состояния; «дешевле контейнеры слабее изоляции» снижает CapEx/OpEx на узел при **осознанном** принятии риска по **модели угроз** — сравнение закладывать как сценарий в TCO, а не как разовый выбор (см. _Приложении D, параграф «[Безопасный MVP контура исполнения за 30 дней, дискуссия по средам и выводы](./20260325-research-appendix-d-security-observability-ru.md#app_d__secure_mvp_execution_environment)»_).

### Около-LLM инфраструктура: журналы, события, нагрузка {: #sizing_near_llm_infrastructure }

Ниже — **статьи OpEx/CapEx** вокруг сервисов, которые отдают LLM/RAG наружу: потоки событий, хранение журналов, нагрузочное тестирование. Иллюстрация переносима на любой стек; источники — открытые описания проектов Ozon Tech.

- **Хранение и поиск журналов:** [seq-db](https://github.com/ozontech/seq-db) позиционируется как масштабируемая СУБД для журналов событий (логов) с одиночным и кластерным режимами — в дереве затрат это **диск, реплики, запросы** и связанный **персонал** сопровождения observability.
- **Пайплайны событий:** [file.d](https://github.com/ozontech/file.d) — сбор и обработка событий (в т.ч. в Kafka, Kubernetes), интеграция с Prometheus; влияет на **OpEx** вычислений и сети при учёте токенов/вызовов в потоках телеметрии (см. также связку file.d → seq-db в документации проектов).
- **Нагрузочное тестирование gRPC:** [framer](https://github.com/ozontech/framer) заявлен как высокопроизводительный генератор нагрузки для gRPC; полезен при проверке **SLO и ёмкости** обвязки инференса. В README проекта указана стадия **alpha** и возможные изменения публичного API — закладывать **риск смены инструментария** в пилоте.

Подробнее переносимые **управленческие** уроки из публичных статей (платформа ботов, поисковый слой, MLOps) — в параграфе _«[Извлекаемые уроки из публичных материалов OZON Tech (РФ)](./20260325-research-report-methodology-main-ru.md#method_ozon_tech_lessons_learned)»_ отчёта «Методология разработки и внедрения ИИ».

### Пре-LLM слой и режимы нагрузки (ориентиры для модели затрат) {: #sizing_pre_llm_layer_load_modes }

**Два режима эксплуатации (аналог маршрутизации моделей в FinOps):** «быстрый путь» — преимущественно детерминированное выделение структурированных идентификаторов без тяжёлого NLP; «полный каскад» — дополнительные стадии для имён, адресов и смешанного языка. На синтетических корпусах IT-поддержки референс-каскад показал **существенно больший recall** против одного regex при **цене** в ложные срабатывания на технический текст без доменных фильтров; в смету закладывают **долю обращений**, проходящих полный путь, и затраты на разметку/поддержку blocklist-политик. Практики **юнит-экономики** и рабочая группа по GenAI — [FinOps Foundation — Generative AI](https://www.finops.org/wg/generative-ai/) и [Unit Economics capability](https://www.finops.org/framework/capabilities/unit-economics/).

**Инженерный ориентир по бэкендам (внутренние замеры референс-стека, не договор к поставщику):** для одной и той же линейки эмбеддингов на ограниченной VRAM корректность pooling и задержка могут отличаться между высокопроизводительным серверным движком и унифицированным мульти-модельным сервисом; **генеративный реранкер** может оставаться на «прямом» пайплайне библиотеки inference, если выбранный сервер не поддерживает требуемый формат скоринга. Это не уникальные числа для КП, а напоминание заложить **отдельные строки** на совместимость, профилирование и регрессии при каждом крупном обновлении.

### Наблюдаемость LLM/RAG: сценарии размещения и бюджет (ориентир для РФ) {: #sizing_llm_rag_observability_budget }

| Сценарий | Типичные статьи затрат | Применимость в РФ (ориентир) |
| :--- | :--- | :--- |
| **Зарубежный SaaS** наблюдаемости | Абонентская плата по объёму трасс/сидов, мало CapEx | Допустим при **явной** правовой оценке, ДПО и допустимости размещения копий данных; для **ПДн** в проде часто **не** дефолт |
| **Self-hosted** (on-prem или IaaS заказчика) | CapEx/аренда ВМ и СХД, занятость специалистов на сопровождение стека | Соответствует типовым ожиданиям **локализации** и контроля журналов при работе с чувствительными данными |
| **Облако РФ** (управляемые ВМ/К8s/SaaS у РФ-провайдера) | OpEx по потреблению + сетевой трафик телеметрии | Компромисс: меньше CapEx, чем свой ЦОД, при сохранении договорного контура РФ |
| **Гибрид** (метаданные в корпорации, контент трасс в зашифрованном хранилище) | Инжиниринг конвейера + отдельное хранилище | Соотносится с рекомендациями **OpenTelemetry** не хранить полный текст промптов в атрибутах спанов по умолчанию ([GenAI spans](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-spans/)) |

Метрики в духе `gen_ai.client.token.usage` и `gen_ai.client.operation.duration` из конвенций **OpenTelemetry GenAI** ([метрики](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-metrics/)) дают **общий язык** с биллингом API (в т.ч. разделение input/output токенов) и упрощают **аллокацию** FinOps по продуктам; методологическая связка наблюдаемости, ПДн и комплекта отчуждения — в _Приложении D, параграф «[Промышленная наблюдаемость LLM, RAG и агентов](./20260325-research-appendix-d-security-observability-ru.md#app_d__llm_rag_agent_observability)»_.

**Иллюстрация переноса CapEx в OpEx (не верифицированная третьей стороной):** в публичных материалах о **MTS AI** и облаке **MWS** приводится оценка **экономии инвестиций порядка свыше 1 млрд руб.** за счёт переноса **обучения и инференса** моделей ИИ в облако провайдера вместо закупки собственной GPU-инфраструктуры ([обзор CNews](https://www.cnews.ru/reviews/provajdery_gpu_cloud_2025/cases/kak_mts_ai_sekonomila_bolee_milliarda), [CIO](https://cio.osp.ru/articles/140525-MTS-AI-perenesla-obuchenie-modeley-v-oblako)). Цифру следует использовать только как **гипотезу** для сравнения сценариев «свой ЦОД / аренда GPU» в финансовой модели заказчика: это **заявление со стороны компании**, методология расчёта и границы TCO в первоисточниках нужно запрашивать при переговорах с поставщиком.

Ниже — **типовое** распределение затрат; фактическое определяется договором (фикс, T&M, BOT, управляемый сервис).

| Вид затрат | Часто у интегратора | Часто у заказчика |
| :--- | :--- | :--- |
| **CapEx** | Лицензии на инструменты разработки, стенды пилота (по соглашению) | Серверы, GPU, СХД, сетевое оборудование, лицензии ОС/ВМ у себя в контуре |
| **OpEx (проект)** | Аналитика, разработка, настройка RAG, интеграции, первичная оценка качества | Внутренний PM/владелец продукта, приёмка, обучение пользователей |
| **OpEx (эксплуатация)** | Поддержка по SLA, доработки по бэклогу (если заказано) | Облачные API или электроэнергия/ЦОД, мониторинг, ИБ, резервное копирование, персонал эксплуатации |
| **Передача (KT / IP)** | Документация, сессии передачи, hypercare по сроку | Владение репозиторием после актов, дальнейшее развитие своими силами |

**Связь с отчуждением:** при передаче «под ключ» заказчик должен заложить OpEx на **своих** специалистов или на продлённую поддержку интегратора; при **BOT** — поэтапный перенос строк бюджета из проекта в линейную эксплуатацию. Детали комплекта передачи и чек-листа приёмки — в параграфе _«[Отчуждение обвязки](./20260325-research-report-methodology-main-ru.md#method_wrapper_detachment)»_ отчёта «Методология разработки и внедрения ИИ» и _Приложении B «[Пакет отчуждения (минимально целостный)](./20260325-research-appendix-b-ip-code-alienation-ru.md#app_b_alienation_package_minimal)»_.

### FinOps и юнит-экономика нагрузки {: #sizing_finops_unit_economics }

Для управляемости бюджета ИИ-нагрузок закрепите **юнит-экономику** в духе практик [FinOps Foundation](https://www.finops.org/wg/generative-ai/).

- **Базовые единицы:** стоимость **на 1 млн токенов**, **на пользователя/месяц**, **на успешный диалог** или **на тикет** — в разрезе среды (облако РФ vs on-prem).
- **Помимо токенов основной модели:** **стоимость пре-LLM шага** (мс на запрос × тариф CPU/GPU или амортизация узла) и **удельный overhead** мульти-бэкенда (число процессов × сопровождение) — отдельной строкой учёта.
- **Связь с P&L:** **аллокация** затрат по продуктам/департаментам и **наблюдаемость** (токены, латентность, ошибки, объём и срок хранения трасс/аудита агентских шагов и вызовов инструментов) — чтобы рост контекста и вызовов инструментов читался в линии P&L.
- **Качество и прод:** связка офлайн/онлайн оценок с метриками прод — _«[LangSmith — Evaluation concepts](https://docs.langchain.com/langsmith/evaluation-concepts)»_.
- **Сверка с биллингом и разрез по слоям:** при инструментировании по [OpenTelemetry GenAI metrics](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-metrics/) учёт токенов по типу (входные/выходные) даёт воспроизводимую основу для сверки с биллингом у провайдера API; метрики времени отклика сервера отделяют узкое место инференса от стоимости поискового слоя в отчётах по юнит-экономике.

Количественную детализацию см. в _«[Модель затрат](#sizing_cost_model)»_; оптимизации (маршрутизация моделей, кэширование промптов, пакетная обработка и размер **батча**) оценивайте через тот же учёт.

**Внешние ориентиры нагрузки:** публичные интервью о клиентском чат-боте — порядки **свыше 3 млн** обращений в месяц, **около 1,7 тыс.** клиентских сценариев, целевая скорость ответа **менее секунды** ([CIO, 2024](https://cio.osp.ru/articles/5455)). Внутренний RAG-помощник для операционистов — пилот на **3 тыс.** сотрудников, заявленная **90%** точность ответов, **девятикратное** сокращение времени поиска ([«Открытые системы», 2025](https://www.osp.ru/articles/2025/0324/13059305)). Эти кейсы задают **порядок величин** для сценарного сайзинга и обсуждения FinOps-метрик. Переносимые паттерны внедрения без привязки к бренду — в параграфе _«[Публично описанные паттерны (финсектор)](./20260325-research-report-methodology-main-ru.md#method_financial_sector_patterns)»_ отчёта «Методология разработки и внедрения ИИ».

Для **внутренней** разработки с агентами (отдельно от **цены за одно обращение в эксплуатации**) полезно фиксировать **стоимость цикла** «план → код → проверка», **число итераций до приёмки** и связанные токены — эти затраты **относят на** центр компетенций или платформу и сверяют с эффектом от упрощения обвязки после обновления модели ([Anthropic — Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps), [Martin Fowler — Harness Engineering](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)).

**Ограничения выборки (OpenAI, *The state of enterprise AI*, декабрь 2025):** цифры — выборка enterprise-клиентов экосистемы OpenAI; для резидентного контура РФ требуется учёт ПДн и тарифов РФ. Рост **токенов рассуждения** ~**×320** за **12** мес. — ориентир нагрузки на «тяжёлые» режимы; в модели разводите учёт чата, рассуждения и агентских циклов, сверяйте метрики потребления токенов с биллингом провайдера. Подробнее о выборке и оргконтексте — в параграфе _«[Эмпирика корпоративного внедрения (отчёт OpenAI, 2025; оговорки по выборке)](./20260325-research-report-methodology-main-ru.md#method_openai_implementation_report)»_ отчёта «Методология разработки и внедрения ИИ».

### Сравнение локального и облачного развертывания {: #sizing_onprem_vs_cloud_comparison }
- **Облако (Cloud):** Старт от **212 500 руб. в месяц**. Подходит для быстрого MVP и малых Pilot.
- **Локально (On-Prem):** Покупка серверов (например, Nvidia H200 может стоить до 50 млн ₽). TCO окупается при высокой нагрузке за счет отсутствия платы за токены. Годовое обслуживание: 10–20% от стоимости железа.

### Метрики эффективности {: #sizing_efficiency_metrics }
- **Цена за тикет:** Стоимость обработки одного обращения ИИ-ассистентом. **Целевое состояние после оптимизаций** (маршрутизация на более дешёвую линейку, кэширование системного промпта, сжатие контекста, доля «лёгкого» контура): ориентир **&lt; 1 ₽** за тикет при сопоставлении с **50–100 ₽** полной стоимости обработки человеком (см. также сборник мер по оптимизации ниже). **Верхняя модельная оценка одного «толстого» цикла** (полный вход с извлечённым контекстом и длинным ответом при усреднённых медианных тарифах за 1 млн токенов) — порядка **2,5–2,7 руб.**; как получены эти **примерные расчёты** и какие допущения заложены — в параграфе _«[Примерные расчёты расхода токенов по данным корпуса заявок (портал поддержки)](./20260325-research-report-sizing-economics-main-ru.md#sizing_token_consumption_estimates)»_ отчёта «Сайзинг и экономика (CapEx / OpEx / TCO)». Для КП всегда пересчитывать через **актуальный прайс** выбранного провайдера и фактический профиль тикетов заказчика.
- **Autoflation Index:** Коэффициент снижения затрат на бизнес-процесс после внедрения ИИ.
- **Точка безубыточности (break-even):** порог утилизации GPU и горизонт владения; количественные сценарии и формулы — в _«[TCO GPU: облако РФ против закупки](#sizing_gpu_cloud_onprem_tco_breakeven)»_. Управленческий ориентир: при **устойчивой** высокой утилизации (**порядка 60–70 %** и выше, плюс горизонт **нескольких лет**) чаще выигрывает закупка/colocation; при низкой или пиковой нагрузке — аренда облака.

## Российский рынок GenAI: зрелость и барьеры {: #sizing_russian_market }

Ключевые метрики российского рынка (сверены с глобальными показателями):

- **85-90%** компаний используют GenAI (глобальный бенчмарк: McKinsey 88%); системно интегрировали — **около трети**.
- **64%** выделяют на GenAI **только 1–5%** бюджета — разрыв между личной зрелостью и корпоративным управлением.
- Барьеры: **40-50%** — качество и галлюцинации; **45-60%** — безопасность и риски утечки данных (Stanford: +56% инцидентов год к году); **53%** — необходимость доработки контента; **49%** — шаблонность.
- **85%** считают GenAI ключевым фактором трансформации на горизонте **трёх лет**.

Полный разбор — в _Приложении E «[Рыночные и технические сигналы](./20260325-research-appendix-e-market-technical-signals-ru.md#app_e_russian_market)»_.

### Учёт токенов рассуждения (reasoning) {: #sizing_reasoning_tokens }

Для моделей с рассуждением (GLM-5, MiniMax-M2.7, Kimi-K2.5, Claude 4.6, GPT-5.4, Gemini 3.1 Pro) фактическая стоимость может превышать базовый расчёт из-за внутренних токенов рассуждения, оплачиваемых по тарифу выходных токенов. Токены рассуждения — это промежуточные шаги вычислений модели, невидимые в ответе, но полностью видимые в биллинге.

| Уровень задачи | Токены рассуждения | Мультипликатор стоимости* | Примеры сценариев |
|:---|---:|---:|:---|
| Без рассуждения | 0 | 1,0× | Простые FAQ, классификация, выборка из базы |
| Лёгкий | 0,5–1× от выхода | 1,5–2,5× | Стандартные инструкции, типовая диагностика |
| Средний | 2–5× от выхода | 4–8× | Технический анализ, многошаговая отладка, юриспруденция |
| Сложный | 8–15× от выхода | 10–20× | Архитектура ПО, сложная математика, новые кейсы |

!!! note "Мультипликаторы стоимости рассуждений"

    Мультипликатор рассчитан относительно базовой модели того же провайдера. Для GLM-5 (Zhipu AI) и MiniMax-M2.7 — самые экономичные рассуждающие модели 2026 — мультипликаторы ниже: 1,2–1,8× (low-medium), 2,0–2,5× (high). Для OpenAI GPT-5.4: 4,9×–6,5×.

!!! tip "Рекомендация по смете"

    При планировании бюджета на агентов с возможностями рассуждения закладывать **средний мультипликатор 5×** к базовому расчёту для неизвестного профиля задач, уточнять до 2× или 10× после замеров на пилоте. Для задач, точно требующих рассуждения (математика, код со сложной логикой), использовать верхнюю границу 10–15×.

Сравнительные **вилки** цен за 1 млн токенов по регионам (США, КНР, РФ) при подготовке оценки дополнительно просматривались через агрегаторы и обзорные публикации; в настоящий документ **не** включены таблицы min/max без повторной сверки с первичными прайсами — для КП использовать только строки раздела «**Тарифы российских облачных провайдеров ИИ**» и договорные ставки.

- Квантование (Q4) снижает требования к VRAM в 4 раза при потере точности < 3%.

## Риски и оптимизация {: #sizing_risks_and_optimization }

### Сборник мер по оптимизации стоимости (Cost Optimization Suite) {: #sizing_cost_optimization_suite }
  1.  **Адаптивная маршрутизация (Model Routing):** Простые вопросы -> SLM (0.6B), сложные -> LLM (30B+). Экономия до 40%. Для self-hosted контура открытые MoE-веса вроде **GigaChat 3.1 Lightning** ([ai-sage/GigaChat3.1-10B-A1.8B](https://huggingface.co/ai-sage/GigaChat3.1-10B-A1.8B): **10B** / **1,8B** активных) могут претендовать на роль быстрого яруса; гипотезу подтверждают замеры latency, throughput и полная стоимость GPU-часа против токенов API, плюс публичные ориентиры вендора ([Хабр](https://habr.com/ru/companies/sberbank/articles/1014146/)). **Политика пре-LLM:** по аналогии закрепляют долю трафика на «лёгком» контуре (детерминированные правила) vs полный NLP-каскад — см. подраздел «Пре-LLM слой и режимы нагрузки» выше.
  2.  **Промпт-кэширование (Prompt Caching):** Использование статических системных промптов. Снижает стоимость входных токенов на 50-80%.
  3.  **Формат TOON (Token-Oriented Object Notation):** Компактная альтернатива JSON для передачи структурированных данных LLM. Снижает потребление токенов на 30-60% при сохранении точности извлечения данных, особенно эффективен для однородных массивов объектов (списки продуктов, пользователей, журналов). См.: [спецификация TOON](https://toonformat.dev/), [Tensorlake: бенчмарки](https://www.tensorlake.ai/blog/toon-vs-json), [Systenics: экономия токенов](https://systenics.ai/blog/2026-01-24-toon-vs-json-how-token-oriented-object-notation-reduces-llm-token-costs).
  4.  **Контекстная компрессия (REFRAG):** Сжатие чанков или удаление нерелевантных данных.
  5.  **Speculative Decoding:** Ускорение инференса большой модели за счет малой модели-предсказателя.

### Требования к VRAM при инференсе LLM {: #sizing_vram_requirements_llm_inference }

### Формула расчёта VRAM {: #sizing_vram_calculation_formula }

**Общий VRAM = Веса модели + KV-кэш + Активации + Оверхед фреймворка**

### Формула оценки {: #sizing_estimation_formula }
```
VRAM ≈ (Параметры × Байт/Вес) / TP + KV-кэш + Оверхед
```

**TP** — число GPU по оси **тензорного параллелизма** (см. _[Приложение A — термины и условные обозначения](./20260325-research-appendix-a-index-ru.md#app_a_conventions_and_scope)_); **TP = 1** соответствует отсутствию такого шардирования (в формуле — деление на единицу).

### Эмпирическое правило {: #sizing_empirical_rule }

- FP16: **~2 GB VRAM на 1B параметров**
- BF16: Аналогично FP16
- INT8: **~1 GB VRAM на 1B параметров**
- INT4: **~0.5 GB VRAM на 1B параметров**

Для грубой прикидки до замеров на целевом стеке полезен внешний калькулятор _«[VRAM calculator — apxml.com](https://apxml.com/tools/vram-calculator)»_; он **не заменяет** учёт **KV-кэша**, **батча** (размера пакета запросов) и оверхеда фреймворка на **инференс на базе vLLM** / **сервер инференса MOSEC** у заказчика. Те же ориентиры (MLCommons, vLLM, калькуляторы VRAM) перечислены в разделе [«Источники» — инференс и VRAM](#sizing_inference_benchmarks_vram_tools).

**Три типа опор для цифр (не смешивайте в одной смете):** (1) **замер на стенде заказчика** — при жёстких SLO; (2) **профиль MOSEC** ниже по документу — типовой эталон вспомогательных моделей **Comindware**, не индивидуальный сайзинг; (3) **публичные профессиональные публикации и энтузиастские обзоры** — только порядок величин для разговора с заказчиком, **не** замер **Comindware** и **не** строка КП **без** отдельного прогона на **его** стеке.

**Колонка «свободно»** в таблицах MOSEC нормирована на **1×24 ГБ**; для **48 ГБ** к остатку грубо **+~24 ГБ** (до своего замера); **96 ГБ** и иные SKU — по факту и прайсу на дату закупки.

### TCO GPU: облако РФ против закупки и порог утилизации {: #sizing_gpu_cloud_onprem_tco_breakeven }

Ниже — **воспроизводимая иллюстрация** для переговоров о модели владения (облако dedicated GPU vs закупка ускорителей): все суммы в **руб.**, пересчёт из USD-рядов прайсов — по _«[валютной политике комплекта](./20260325-research-appendix-a-index-ru.md#app_a_fx_policy)»_. **Не** смешивайте в одной строке сметы: облачный ряд — **операционная аренда** по публичным тарифам; on-prem ряд — **CapEx закупки GPU** из [матрицы цен](#sizing_gpu_hardware_pricing_all) без полного учёта ЦОД, сети, персонала и налоговой амортизации (их заказчик добавляет в своей модели).

**Допущения сценария «непрерывная нагрузка»:** **730 ч/мес** на весь период (**8 760 ч/год** на GPU) — типовой множитель из прайса Cloud.ru; горизонт **3 года** → **36** таких месяцев. Для спайковой нагрузки умножьте часы фактической занятости.

**Шаг 1 — облако (8× H100, РФ):** в открытом прайсе Cloud.ru нет строки ровно **8×**; ближайшая опора — **7×H100 NVLink** (**~5 978 ₽/ч** за конфигурацию, см. [таблицу Cloud.ru](#sizing_cloudru_gpu)). Эквивалент **8 GPU** — линейное масштабирование ставки: **(8/7) × 5 978 = 6 832 ₽/ч**. Тогда

- **3 года:** 6 832 × 730 × 36 = **179 544 960** руб. (округлённо **~179,5 млн**).

**Альтернативная вилка по публичным ориентирам «за 1× H100»** (Selectel, см. [таблицу Selectel](#sizing_selectel_gpu)): **~900–2 200 ₽/ч** на GPU → за 3 года при **730 ч/мес**:

- **низ:** 900 × 8 × 730 × 36 ≈ **189 216 000** руб. (**~189 млн**);
- **верх:** 2 200 × 8 × 730 × 36 ≈ **462 528 000** руб. (**~463 млн**).

**Шаг 2 — закупка GPU (только карты, 8× H100 80GB):** из [той же матрицы](#sizing_gpu_hardware_pricing_all) диапазон **~1 290 000 – 2 550 000** руб. за карту → **~10 320 000 – 20 400 000** руб. **единовременного CapEx** на восемь ускорителей (без серверного шасси и прочего).

**Шаг 3 — сопоставление:** приведённый CapEx **~10–20 млн** относится к **закупке**; ряд **~179–463 млн** — к **трёхлетней** аренде при **полной** занятости по часам выше. Отсюда **не** следует один коэффициент «экономии 70 %» без профиля часов: при снижении фактических часов аренды облако выигрывает; при **высокой** устойчивой утилизации и учёте только «цены железа» выигрывает закупка. Полный on-prem TCO заказчик собирает из CapEx узла, электроэнергии, обслуживания и риска устаревания — см. _«[Локальное развёртывание — крупномасштабное](#sizing_large_scale_enterprise)»_ и _«[Амортизация и жизненный цикл](#sizing_depreciation_hardware_lifecycle)»_.

**Ориентир по одной карте (облако, доля от пакета Cloud.ru):** для **5×H100** в прайсе **~2 745 ₽/ч** на весь узел → **549 ₽/ч** на одну долю GPU при грубом делении; за год при 8 760 ч: **549 × 8 760 = 4 809 240** руб. (округлённо **~4,81 млн руб./год** на условный «1 GPU-эквивалент» из этой строки каталога). Это **ниже** верхней вилки Selectel и показывает разброс между провайдерами и SKU.

**Порог утилизации (управленческий ориентир, не норма РФ):** в обзорной работе по экономике развёртываний ИИ на структурированных публичных данных сформулированы эвристики: при **низкой** утилизации выгоднее облако; **устойчивая** загрузка **выше ~60–70 %** и горизонт **нескольких лет** сдвигают баланс в сторону собственной инфраструктуры или colocation; параметрически порог **3-летнего** break-even по GPU встречается в коридоре **примерно 55–75 %** в зависимости от тарифа, электроэнергии и графика амортизации ([Ivchenko, 2026](https://doi.org/10.5281/zenodo.18678386) — обзор моделей TCO и ссылка на рамку Lawrence Berkeley National Laboratory в тексте статьи). Для резидентного контура РФ обязательно пересчитать **практическим** прайсом выбранного провайдера и курсом/договором на дату сметы.

### OpEx безопасности GenAI и агентов (переменная статья) {: #sizing_genai_security_opex }

В TCO промышленного ассистента закладывают не только железо и токены, но и **постоянный контур проверки**: периодический **AI red teaming** (внутренний или внешний отчёт), обновление сценариев под [OWASP LLM Top 10 2025](https://genai.owasp.org/resource/owasp-top-10-for-llm-applications-2025/) и при наличии инструментов — [OWASP Agentic Top 10 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/), прогон открытых сканеров вроде [Garak](https://github.com/NVIDIA/garak) на изолированных стендах, обучение разработчиков и линии поддержки **безопасной работе с GenAI** (фишинг против пользователя и «промпт против модели»). Суммы в смете не фиксируем здесь: они зависят от масштаба, требований регулятора и выбора подрядчика; при планировании закладывают **постоянную занятость специалистов ИБ и машинного обучения и периодические закупки услуг** так же, как на классический пентест.

**Контекст рынка (не обязательный поставщик):** в деловой прессе отмечают рост внимания к атакам на ИИ-системы и смежным услугам ([Коммерсантъ](https://www.kommersant.ru/doc/8363105)); крупные вендоры усиливают направления тестирования GenAI (иллюстрация консолидации — [OpenAI — приобретение PromptFoo](https://openai.com/index/openai-to-acquire-promptfoo/)). Для заказчика это сигнал **дефицита кадров и инструментов**, а не указание закупить конкретный продукт. Детализация чек-листов и ссылок — в _Приложении D, параграф «[Безопасность генеративного ИИ, OWASP и сигналы рынка](./20260325-research-appendix-d-security-observability-ru.md#app_d__genai_security_owasp_tco_risks)»_.

**Класс OpEx AI TRiSM:** в терминологии [Gartner — AI TRiSM](https://www.gartner.com/en/information-technology/glossary/ai-trism) (в ряде обзорных материалов встречается написание «AI-TRISM») к **доверию к ИИ** относят объяснимость, защиту моделей и данных, соблюдение требований и устойчивость операций. В смете промышленного GenAI закладывают работы по **интерпретируемости**, контуры **ModelOps** под гардрейлы и аудит, периодические проверки на соответствие политикам и сопровождение после смены модели — **рядом** с уже описанным OpEx red teaming / OWASP и **не** заменяя его. Углубление — _Приложение D, параграф «[AI TRiSM и управление доверием](./20260325-research-appendix-d-security-observability-ru.md#app_d__ai_trism_trust_management)»_.

|-------------|--------|--------------|

### Дополнительные стратегии оптимизации затрат {: #sizing_additional_cost_optimization_strategies }

1.  **Выбор моделей:** Используйте модели 0.6B для приложений с чувствительностью к стоимости; резервируйте 4B+ для высокоточных задач.
2.  **Смешанная точность:** Используйте fp16/bf16 для уменьшения VRAM на 50% по сравнению с fp32.
3.  **Обслуживание нескольких моделей:** MOSEC позволяет развертывать несколько моделей на одном сервере, снижая потребности в оборудовании.
4.  **Прерываемые инстансы:** Yandex Cloud и другие российские провайдеры предлагают прерываемые (preemptible) GPU-инстансы со скидкой до 40–60% для некритичных рабочих нагрузок.
5.  **Автоматическое масштабирование:** Масштабируйте облачные ресурсы в зависимости от спроса для минимизации простоев.

### Инференс и VRAM: бенчмарки, vLLM и калькуляторы {: #sizing_inference_benchmarks_vram_tools }

#### Ориентиры для глубокого аппаратного сайзинга (официальные бенчмарки и документация) {: #sizing_hardware_deep_research_pointers }

Для глубокого аппаратного сайзинга опирайтесь на **официальные** и воспроизводимые источники: отраслевые бенчмарки **MLCommons**, документацию и release notes **vLLM**, а также VRAM-калькуляторы как вспомогательный инструмент для грубой прикидки. Эти источники помогают не подменять инженерную проверку маркетинговыми цифрами и отделять оценку **железа**, **KV-кэша**, **батча** и **оверхеда фреймворка**.

#### Бенчмарки RTX 4090 (24 и 48 ГБ) {: #sizing_rtx_4090_benchmarks }

В этом отчёте **RTX 4090** используется как ориентир в двух вариантах: **24 ГБ** для локальных пилотов и рабочих станций и **48 ГБ** для более ёмких локальных и арендных конфигураций. Для решений по CapEx / аренде смотрите вместе профиль **on-prem-GPU**, пропускную способность памяти и типовые модели для локального инференса; юридические и лицензионные нюансы по поставке GPU вынесены в _Приложение B_.
