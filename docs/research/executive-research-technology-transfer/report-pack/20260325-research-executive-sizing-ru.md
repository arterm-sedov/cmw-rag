# Резюме для руководства: сайзинг, CapEx и OpEx (C-level) {: #exec_sizing }

## Решение за 60 секунд {: #exec_sizing_decision_card }

- **Рекомендуемый путь:** облачный PoC в РФ → переход к гибриду/on-prem при стабильной утилизации GPU >60 %.
- **Бюджетный ориентир:** использовать вилки как порядок величин; договорная смета только после проверки прайса на дату и стендовых замеров.
- **Ключевой риск:** перенос глобальных бенчмарков в КП без учета требований ПДн/резидентности и локальных тарифов.
- **Следующий шаг:** зафиксировать 30-дневный план измерений (нагрузка, качество, эскалации, стоимость токена/сценария).

## Как использовать это резюме {: #exec_sizing_how_to_use }

**Для обоснования инвестиций:**

- Утвердить бюджетные диапазоны (CapEx/OpEx/TCO), сравнение облако РФ / on‑prem / гибрид и правила пересчёта/чувствительности.

**Для переговоров с C-Level:**

- Использовать вилки как «порядок величин» и переносить в КП только после сверки прайса на дату и (для on‑prem) замеров на целевом стеке/SLO.

!!! warning "Ограничения"

    Не используйте отчёт как:

    - Точную смету без замеров на стенде заказчика (глобальные бенчмарки — контекст рынка, не норма для КП).
    - Базовую конфигурацию для резидентного продакшна без учёта требований по ПДн и тарифов РФ.

!!! note "Границы применимости"

    Глобальные бенчмарки (OpenAI, a16z, McKinsey) — это **контекст рыночных трендов**, а не договорная норма для российских КП. Финальные цифры формируются только после сверки прайса на дату и, для on-prem, после стендовых замеров.

## Ключевые тезисы для принятия решений {: #exec_sizing_key_theses }

**Финансовая база решения.** Оценка строится на публичных тарифах облаков РФ, открытых бенчмарках и профиле on-prem-GPU по материалам Comindware. Для бюджета и КП применяются только значения, подтверждённые на дату расчёта и на целевом стеке заказчика.

**Драйверы стоимости и сценарии.** Экономика рассчитывается по пяти блокам: токены, инфраструктура, интеграция, сопровождение, безопасность/eval. Смена модели и режима работы (чат, reasoning, агентские циклы) существенно меняет итоговый TCO и должна считаться отдельными сценариями.

**Выбор модели владения (SaaS / on-prem / гибрид).** Сравнение выполняется в едином формате неопределённости: вилки и разделение разовых/повторяющихся затрат. Для управленческого решения по модели владения используются только сопоставимые строки затрат и единые допущения; детальные формулы — в _[основном отчёте по сайзингу](./20260325-research-report-sizing-economics-main-ru.md)_.

**GPU как бизнес-ёмкость.** Планирование начинается с топологии мощности (1×24 ГБ, 1×48 ГБ, 1×96 ГБ, multi-GPU, multi-node), а не с бренда модели. Это снижает риск неверного CapEx и помогает выбрать архитектуру под целевые SLO и темп роста нагрузки.

**Рыночные сигналы РФ.** Сегментные ориентиры применяются как контекст спроса и рисков, с обязательной проверкой формулировок показателей в первоисточниках. Показатели по разным типам риска (например, галлюцинации и утечки) учитываются раздельно в финансовой и риск-модели.

**Глобальные метрики в локальном контуре.** Данные международных enterprise-отчётов используются для проверки чувствительности модели, но не как применимая напрямую база для российских КП. Финальное решение принимается с учётом требований ПДн, резидентности и локальных тарифов.

## Целевые метрики (Benchmark) {: #exec_sizing_benchmarks }

- **Утилизация:** >60 % сотрудников используют ИИ ежедневно.
- **Эффективность:** сокращение времени на решение задачи на 30–40 %.
- **Качество:** >95 % ответов соответствуют внутренней рубрике (LLM-as-judge).
- **TCO Break-even:** переход on-prem оправдан при утилизации GPU >60 %.
