## Риски бюджета и меры снижения

| Риск | Влияние | Мера снижения |
| :--- | :--- | :--- |
| **Context Bloat** | Высокое (рост OpEx) | Жесткие лимиты токенов, семантическое сжатие истории. |
| **Дефицит GPU (санкции)** | Среднее (рост CapEx) | Использование российских облаков, переход на CPU-инференс (BitNet). |
| **Галлюцинации** | Высокое (репутация) | Внедрение Guardrails, многоуровневая верификация. |
| **Регрессии стека инференса** | Среднее (качество, искажение бенчмарков) | Пиновать версии vLLM/SGLang; баг SGLang при `dp > 1` ([PR #18802](https://github.com/sgl-project/sglang/pull/18802), [Хабр](https://habr.com/ru/companies/sberbank/articles/1014146/)); для GigaChat на HF сверять карточку на **минимальные коммиты** движка при tool calling ([пример — Lightning 3.1](https://huggingface.co/ai-sage/GigaChat3.1-10B-A1.8B)); регрессионные eval перед обновлением. |
| **Мульти-бэкенд и несовместимость форматов** | Среднее (рост CapEx/OpEx на узлы, задержки релизов) | Единая матрица «модель → поддерживаемый сервер → тест корректности»; закладывать FTE на сопровождение нескольких процессов; при консолидации проверять pooling/форматы реранкеров. |
| **Расхождение экспертных диапазонов в СМИ и внутренней модели** | Низкое–среднее (ошибка бюджетирования) | Таблица **«Матрица принятия решений и экономика (РБК 2026)»** в сопутствующем резюме **Методология внедрения и отчуждения ИИ** согласована с разделами **CapEx/OpEx Модель** и **TCO** настоящего документа; при новых публикациях пересчитывать только через **прайс провайдера** и **собственные допущения**. |
| **Композитный инцидент (веб/API + RAG + политики)** | Высокое (утечки, простой, репутация) | Разделять хранилища политик, секретов и данных; ASVS/WSTG для шлюза; сценарии по [OWASP LLM Top 10 2025](https://genai.owasp.org/resource/owasp-top-10-for-llm-applications-2025/) и при агентах — [Agentic 2026](https://genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026/); ориентир по классу ошибок — [разбор red team (CodeWall)](https://codewall.ai/blog/how-we-hacked-mckinseys-ai-platform). |
| **Теневые ИИ-инструменты и социальная инженерия** | Среднее (обход ИБ, вредонос под видом «клиента ИИ») | Политика допустимых моделей и каналов установки; обучение пользователей; мониторинг конечных точек по регламенту заказчика; внешний **бенчмарк** по росту угроз под видом популярных ИИ-сервисов — [Kaspersky (пресс-релиз, 2025)](https://www.kaspersky.com/about/press-releases/kaspersky-chatgpt-mimicking-cyberthreats-surge-115-in-early-2025-smbs-increasingly-targeted). |
| **Иллюзия QA у LLM-only evaluator** | Среднее–высокое (дефекты в прод, повторные циклы) | Жёсткие пороги по критериям, few-shot калибровка, **инструментальные и e2e** тесты вместо одной самооценки модели ([Anthropic — Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps), [Martin Fowler — Harness Engineering](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)). |
| **Перегруженный harness после апгрейда модели** | Среднее (лишние токены, латентность процесса) | Периодически **упрощать** оркестрацию и проверять, какие шаги ещё «несут нагрузку» по качеству ([Anthropic — Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)). |
| **Рост OpEx от артефактов проверки** (скриншоты, логи прогонов) | Среднее (СХД, ПДн) | Политика **минимизации**, ретенция и контур хранения согласованно с 152-ФЗ; см. сопутствующее резюме **Методология внедрения и отчуждения ИИ** (телеметрия и периметр до LLM). |
