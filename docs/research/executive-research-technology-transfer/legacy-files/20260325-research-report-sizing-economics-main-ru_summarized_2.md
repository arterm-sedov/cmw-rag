# Сайзинг и экономика (CapEx / OpEx / TCO) {: #sizing }

## Обзор {: #sizing_overview }

**Финансовая база** для обоснования инвестиций в ИИ: тарифные сетки российских облачных провайдеров, диапазоны CapEx/OpEx, дерево факторов стоимости и сценарный анализ TCO для различных архитектурных моделей.

**Практический смысл:** обеспечение CFO/CIO доказательной базой для выбора модели владения и защиты TCO-обоснования. Приведенные показатели являются оценочными диапазонами для принятия стратегических решений. Финальные бюджетные обязательства фиксируются после верификации нагрузочного профиля.

Для переговоров и управленческих брифов: переносите бюджетные вилки и пороги утилизации, не отдельные технические таблицы.

Для составления смет проводите замеры на стенде заказчика и получайте актуальные прайсы провайдеров.

## Концепция финансовой модели {: #sizing_scqa }

- **Ситуация:** совокупная стоимость владения (TCO) GenAI-системами формируется из затрат на токены, инфраструктуру GPU, интеграционные работы и непрерывное сопровождение. Профиль нагрузки **Comindware** определяется взаимодействием корпоративного RAG-контура и агентного слоя платформы.
- **Проблема:** стоимость вспомогательных моделей (эмбеддинг, реранкинг, модерация) и мульти-бэкенд архитектур часто недооценивается. Колебания валютных курсов и скрытые статьи (безопасность, регрессионное тестирование) критически влияют на выбор между облачной арендой и собственным контуром.
- **Задача:** определение оптимальной модели финансирования (Cloud&nbsp;RF / On-prem / Hybrid) и расчет целевых диапазонов CapEx/OpEx на горизонте 1–3&nbsp;лет.
- **Решение:**
    - Инициируйте внедрение с облачного PoC.
    - Переход к оценке безубыточности on-prem целесообразен при устойчивой утилизации **>60%** и горизонте планирования от 3&nbsp;лет.
    - Используйте сценарный сайзинг, включая в OpEx затраты на наблюдаемость (накопление трасс, аудит) и пре-LLM обработку (маскирование ПДн, предиктивная фильтрация).
    - Пи измерении бизнес-эффекта учитывайте организационную зрелость.

### Управленческие компромиссы {: #sizing_management_tradeoffs }

Выбор модели финансирования определяет долгосрочный TCO и операционную гибкость — ключевые компромиссы по каждому варианту:

- **SMB / департамент** — низкий CapEx, оплата по токену vs меньше контроля над данными и вендор-лок на API.
- **Enterprise** — инвестиция в GPU и команду LLMOps vs снижение долгосрочного TCO и суверенитет.
- **Гибрид** — гибкость vs сложность учёта и сопровождения двух контуров наблюдаемости.
- **Открытые веса (MIT) vs управляемый API** — без счётчика токенов vs CapEx/аренда GPU и инженерия; вилка разобрана в разделе «**Открытые веса и API: влияние на TCO**» ([описание — Хабр, Сбер](https://habr.com/ru/companies/sberbank/articles/1014146/)).

**Примеры метрик:** стоимость 1 млн токенов по выбранному провайдеру, утилизация GPU, полные 3-летние TCO on-prem vs cloud, чувствительность к курсу и пошлинам.

### Ролевой фокус ЛПР {: #sizing_c_role_decisions }

Сводная матрица приоритетов по ролям ЛПР — _[Стратегическое резюме: матрица аргументов](./20260331-research-executive-unified-ru.md#exec_unified_role_matrix)_.

Финансовые акценты для решения по модели владения и бюджету:
- **CFO:** границы CapEx/OpEx, пороги утилизации, чувствительность к курсу и прайсу.
- **CIO/CTO:** стоимость архитектурных вариантов (cloud/on-prem/hybrid), эффект на SLA и операционную нагрузку.
- **CEO/CRO/CPO/CISO:** экономические последствия выбранной модели владения и ограничений комплаенса.

### Контрольные метрики для инвестиционного решения {: #sizing_figure_validation_registry }

| Метрика | Значение/диапазон | Что это значит для решения |
| :--- | :--- | :--- |
| Порог перехода к оценке on-prem | >60% утилизации | При устойчивой утилизации **сравнивайте** on-prem и cloud по полной стоимости владения |
| Ориентир SaaS (рынок, не смета) | ~200–250&nbsp;тыс. руб./мес | **Используйте** как рыночный бюджетный коридор для этапа PoC/Pilot |
| Ориентир On-prem CapEx узла | ~7–11&nbsp;млн руб. | **Оценивайте** по нему входной инвестиционный порог |
| Ориентир 3-летнего break-even по GPU | ~55–75% утилизации | Используйте как управленческий коридор для сравнения cloud vs on-prem |

Решения по бюджету и модели владения принимаются на основании этих метрик. Отклонения допустимы только при явной фиксации причины и даты пересчёта.

Российский рынок GenAI значительно меньше глобального, но растёт опережающими темпами благодаря регуляторной поддержке и локализационным требованиям.

- **Концентрация рынка:** Top-5 игроков (Яндекс, Сбер, Т-Технологии, ВК, Касперский) контролируют **~95% выручки** AI-сегмента.
- **Барьеры внедрения:** **40–50%** компаний отмечают проблемы качества и галлюцинаций; **45–60%** выражают опасения по поводу утечки данных и безопасности (Stanford: +56,4% инцидентов год к году).

## Экономический каркас и модель затрат {: #sizing_economic_framework .pageBreakBefore }

Стоимость ИИ-решения складывается из трех уровней:
1. **Инфраструктура:** Вычисления на GPU, хранилище векторных БД (ChromaDB, pgvector), Сеть.
2. **Инференс:** Стоимость токенов, эмбеддинги, реранкинг, слой перед LLM (DLP, маскирование), перерасход памяти фреймворков.
3. **Эксплуатация:** LLMOps, мониторинг (Langfuse, Phoenix), хранение трасс, обновление моделей, регулярная синхронизация док ↔ код.

### Около-LLM инфраструктура: журналы, события, нагрузка {: #sizing_near_llm_infrastructure }
Ниже — статьи OpEx/CapEx вокруг сервисов, которые отдают LLM/RAG наружу:
- **Хранение и поиск журналов:** seq-db (СУБД для логов) — влияет на диск, реплики и персонал.
- **Пайплайны событий:** file.d (интеграция с Kafka, Prometheus) — влияет на OpEx вычислений.
- **Нагрузочное тестирование:** framer (gRPC).

### FinOps и юнит-экономика нагрузки {: #sizing_finops_unit_economics }
Для управляемости бюджета ИИ-нагрузок закрепите **юнит-экономику**:
- **Базовые единицы:** стоимость на 1 млн токенов, на успешный диалог или на тикет.
- **Себестоимость транзакции (Unit Cost):** Оптимизированное состояние: **< 1&nbsp;руб.** за инцидент. Потолок затрат «тяжёлого» цикла: порядка **2,5–2,7&nbsp;руб.**

### Топология ёмкости и технический сайзинг {: #sizing_capacity_topology }

**Формула расчёта VRAM:** Общий VRAM = Веса модели + KV-кэш + Активации + Перерасход фреймворка.
*Эмпирическое правило (на 1B параметров):* FP16/BF16: ~2 ГБ; INT8: ~1 ГБ; INT4: ~0,5 ГБ.

**Перерасход памяти фреймворков (vLLM vs MOSEC):** vLLM имеет высокий overhead из-за KV-кэша. Вспомогательные модели (Embedder + Reranker + Guard, все по 0.6B) в сумме занимают **~10–15 ГБ в vLLM**, но всего **~5 ГБ в MOSEC**. Это позволяет безопасно размещать их на картах 24 ГБ вместе с основной моделью.

**Оценка пропускной способности (Throughput):** Макс. ток/cек ≈ Пропускная способность памяти (ГБ/с) / Размер модели (ГБ). 
Например, RTX 4090 (1 008 ГБ/с) для модели 7B (Q4) выдаст ~288 ток/c. H100 (3 352 ГБ/с) для 70B (Q4) выдаст ~96 ток/c.

## Детальные затраты и сценарный TCO (3 года) {: #sizing_detailed_costs_tco .pageBreakBefore }

### Повторяющиеся затраты (OpEx) {: #sizing_recurring_costs }
- **Зарубежные API (разработка):** 4 250 – 42 500 руб./мес (агрегаторы, OpenRouter) для песочниц.
- **Электричество:** RTX 4090 (~400 Вт) = ~4 250 – 8 500 руб./мес; A100 сервер (~2000 Вт) = ~25 500 – 42 500 руб./мес.
- **Поддержка и обслуживание:** Обновление моделей, бэкапы, ИБ-патчи оцениваются в 425 000 – 1 275 000 руб./год.
- **OpEx безопасности (AI TRiSM):** В TCO закладывается AI red teaming, тесты на промпт-инъекции (OWASP LLM Top 10) и ModelOps. Скрытые затраты на расследование инцидента часто превышают годовой бюджет на токены.

### TCO GPU: облако РФ против закупки и порог утилизации {: #sizing_gpu_cloud_onprem_tco_breakeven }
Аренда **8× H100** в облаке РФ (непрерывно, 3 года) обойдется в **~179–463 млн руб** (в зависимости от провайдера). Закупка самих карт (CapEx) составит **~10,3–20,4 млн руб**. 

*Управленческий вывод:* При устойчивой загрузке **выше ~60–70 %** и горизонте 3 лет, собственная инфраструктура или colocation безоговорочно экономически выгоднее аренды.

### Сценарии развёртывания и оценка TCO (на 3 года) {: #sizing_tco_scenarios }
Данные сценарии объединяют рекомендуемые конфигурации, капитальные затраты и 3-летний TCO.

#### 1. Малый бизнес / Департамент (1–3 пользователя)
- **Профиль:** Разработка, тестирование, малые команды, агенты на моделях 0,6B–32B.
- **Оборудование On-Prem:** 1–2× RTX 4090 (24/48 ГБ).
- **CapEx On-Prem:** ~400 000 – 600 000 руб.

## Риски, техническая оптимизация и тренды инференса {: #sizing_risks_optimization }

### Сборник мер по оптимизации стоимости (Cost Optimization Suite) {: #sizing_cost_optimization_suite }
1. **Адаптивная маршрутизация (Model Routing):** направляйте простые вопросы на SLM (**0,6B**), сложные — на LLM (**30B+**); экономия до **40%**.
2. **Формат TOON (Token-Oriented Object Notation):** компактная альтернатива JSON. Снижает потребление токенов на 30–60% при сохранении точности извлечения данных.
3. **Кэширование промптов:** статические системные промпты снижают стоимость входных токенов на 50–80%.
4. **Упреждающая генерация:** ускорение инференса большой модели за счёт малой модели-предсказателя.
