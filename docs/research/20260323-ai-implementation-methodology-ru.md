## 1. Обзор

В данном документе описывается методология внедрения и управления инфраструктурой ИИ в экосистеме CMW с фокусом на российские облачные провайдеры и локальный инференс. Архитектура основана на **модульном, контейнеризованном подходе**, объединяющем RAG-движок (cmw-rag) с специализированными серверами инференса (cmw-mosec, cmw-vllm) и интеграцией с российскими облачными платформами.

**Ключевые методологические принципы:**
- **Разделение ответственности:** Отдельные слои для обработки данных, поиска, инференса и доставки API.
- **Гибридный поиск:** Объединение векторного поиска (плотного) с ключевым (разреженным) для оптимальной точности.
- **Агентная архитектура:** Использование агентов LangChain для динамического вызова инструментов и рассуждения.
- **Гибкость инфраструктуры:** Поддержка как MOSEC (единый сервер), так и vLLM (распределенные инстансы) бэкендов инференса.
- **Российская суверенность:** Приоритет российских облачных провайдеров (Cloud.ru, Yandex Cloud, SberCloud) для обеспечения соответствия требованиям о данных и инфраструктуре.

## 2. Архитектура внедрения

### 2.1 Основные компоненты

| Компонент | Репозиторий | Роль | Технология |
|-----------|------------|------|------------|
| **RAG-движок** | `cmw-rag` | Оркестрация поиска, генерации и логики агентов | Python, LangChain, Gradio |
| **Сервер инференса (Унифицированный)** | `cmw-mosec` | Обслуживание моделей эмбеддинга, реранкера и охранника на одном порту | MOSEC, PyTorch |
| **Сервер инференса (Распределенный)** | `cmw-vllm` | Обслуживание LLM и пулинг-моделей через vLLM | vLLM, CUDA |
| **Векторное хранилище** | `cmw-rag` | Постоянное хранение встраиваний документов | ChromaDB (HTTP) |

### 2.2 Поток данных и конвейер

1.  **Ингестия:**
    *   Документы (Markdown, MkDocs) обрабатываются `rag_engine/core/document_processor.py`.
    *   Разбиваются на чанки через `rag_engine/core/chunker.py` (токен-зависимые).
    *   Встраиваются через `rag_engine/retrieval/embedder.py` (FRIDA/Qwen3).
    *   Хранятся в ChromaDB через `rag_engine/storage/vector_store.py`.

2.  **Поиск (RAG):**
    *   Пользовательский запрос поступает в `rag_engine/retrieval/retriever.py`.
    *   **Векторный поиск:** ChromaDB извлекает top-k чанков.
    *   **Реранкинг:** Кросс-энкодер или LLM-реранкер (`rag_engine/retrieval/reranker.py`) уточняет результаты.
    *   **Сборка контекста:** Статьи восстанавливаются, при необходимости суммируются (`rag_engine/llm/summarization.py`).

3.  **Генерация:**
    *   **Режим агента (Рекомендуется):** Агент LangChain анализирует запрос, вызывает инструмент `retrieve_context` (форсировано через `tool_choice`) и генерирует ответ с цитатами.
    *   **Прямой режим:** Менеджер LLM (`rag_engine/llm/llm_manager.py`) генерирует ответ напрямую из найденного контекста.

4.  **Доставка:**
    *   **Веб-интерфейс:** Gradio ChatInterface (`rag_engine/api/app.py`).
    *   **API:** REST-эндпоинт `/api/query_rag`.
    *   **Виджет:** Встраиваемый HTML/JS виджет (`ui/gradio-embedded.html`).

### 2.3 Конфигурация сервера инференса

**Вариант А: MOSEC (Унифицированный сервер)**
*   **Команда:** `cmw-mosec serve`
*   **Порт:** 8001 (по умолчанию)
*   **Модели:** Эмбеддинг, Реранкер, Охранник загружаются динамически.
*   **Плюсы:** Единый процесс, один порт, эффективное разделение ресурсов.
*   **Минусы:** Ограничено моделями, поддерживаемыми раннером MOSEC.

**Вариант Б: vLLM (Распределенные инстансы)**
*   **Команда:** `cmw-vllm start --model <model_id> --port <port>`
*   **Порты:** Отдельные порты для каждой модели (например, 8100, 8101, 8105).
*   **Модели:** LLM, Эмбеддинг (Qwen3), Реранкер, Охранник.
*   **Плюсы:** Поддержка оптимизаций vLLM (KV-кэш, непрерывная пакетная обработка), гибкий выбор моделей.
*   **Минусы:** Более высокий оверхед VRAM, несколько процессов.

## 3. Методология отчуждения ИИ

### 3.1 Отчуждение данных
*   **Поддержка ChromaDB:** Скрипты `maintain_chroma.py` и `inspect_db_schema.py` позволяют диагностировать, очищать и мигрировать.
*   **Удаление векторного хранилища:** Коллекции можно удалить через HTTP API ChromaDB или Python-клиент.
*   **Архивация документов:** Исходные документы (Markdown) остаются в файловой системе; векторные данные не теряются, если сохранен источник.

### 3.2 Отчуждение моделей
*   **Обновление конфигурации:** Смена ID моделей в `.env` или `config/models.yaml`.
*   **Горячая перезагрузка:** MOSEC поддерживает динамическую загрузку/выгрузку моделей (для vLLM требуется перезапуск).
*   **Версионирование:** Модели отслеживаются через HuggingFace Hub; откат изменением ID модели.

### 3.3 Отчуждение инфраструктуры
*   **Завершение процесса:** `cmw-mosec stop` или `cmw-vllm stop`.
*   **Завершение контейнера:** При развертывании через Docker стандартные команды остановки/удаления контейнеров.
*   **Очистка ресурсов:** Память GPU освобождается при завершении процесса; данные ChromaDB сохраняются на диске до ручного удаления.

## 4. Рекомендации по производственной эксплуатации (2026)

На основе исследования «Продвинутые подходы к RAG»:

1.  **Гибридный поиск:** Реализуйте BM25 + Плотный поиск для точности уровня enterprise (4-7,5% прирост).
2.  **Адаптивная маршрутизация:** Используйте анализ сложности запроса для прямой маршрутизации простых запросов в LLM, избегая ненужного поиска.
3.  **Самокоррекция:** Реализуйте механизмы критики для сложных запросов для уменьшения галлюцинаций.
4.  **Мониторинг:** Отслеживайте точность поиска, релевантность контекста и частоту галлюцинаций.

## 5. Рекомендации

1.  **Для новых внедрений:**
    *   Начните с `cmw-mosec` для простоты (единый сервер).
    *   Используйте режим агента в `cmw-rag` для динамического вызова инструментов.
    *   Реализуйте гибридный поиск (BM25 + Вектор) для оптимальных результатов.

2.  **Для масштабирования:**
    *   Переходите на `cmw-vllm` для инференса LLM (лучшая производительность).
    *   Используйте отдельные инстансы vLLM для эмбеддинга/реранкера/охранника для распределения нагрузки.
    *   Рассмотрите Kubernetes для оркестрации при масштабировании на несколько узлов.

3.  **Для отчуждения:**
    *   Архивируйте исходные документы перед удалением векторных данных.
    *   Используйте `maintain_chroma.py` для диагностики состояния базы данных перед выключением.

## 7. Архитектуры RAG нового поколения от канала «Эй ай надзор»

На основе материалов канала **@ai_archnadzor** (практикующий AI-архитектор):

### 7.1 Disco-RAG: Логический анализ вместо «плоского супа» из фактов [[источник]](https://t.me/ai_archnadzor/183)

**Концепция:** Внедрение теории риторических структур (RST). Модель понимает, где аргумент, где противоречие, где условие.

**Архитектура:**
*   **Intra-chunk RST Trees:** Для каждого чанка строится дерево связей (Nucleus/Satellite)
*   **Inter-chunk Rhetorical Graph:** Анализ отношений между чанками (дополняет/противоречит)
*   **Discourse-Aware Planning:** План ответа на основе графа связей перед генерацией

**Результат:** Превращает RAG из «читателя фактов» в «аналитика логики»

### 7.2 REFRAG: Ускорение RAG в 30 раз [[источник]](https://t.me/ai_archnadzor/178)

**Проблема:** Огромный контекст убивает TTFT и «съедает» KV-Cache

**Решение:** Сжатие «сырых» чанков в компактные эмбеддинги через RoBERTa + селективное расширение через RL-политику

**Для кого:** Tier-1 системы с миллионами запросов, где важна скорость

### 7.3 Cog-RAG: Гиперграфы и «тематическое» мышление [[источник]](https://t.me/ai_archnadzor/179)

**Концепция:** Двойные гиперграфы (темы и сущности) для имитации человеческого подхода «от общего к частному»

**Результат:** Win Rate выше на **84.5%** по сравнению с обычным RAG

**Вердикт:** Мощно, но дорого по индексации. Идеально для медицины и науки

### 7.4 HippoRAG 2: Экономим на графах в 12 раз [[источник]](https://t.me/ai_archnadzor/180)

**Инновация:** Dual-Node архитектура (узлы-сущности + узлы-пассажи)

**Экономика:** Снижение затрат на токены при индексации в **12 раз** (9 млн токенов vs 115 млн)

**Стек:** `pip install hipporag`

### 7.5 Topo-RAG: Победа над «табличной слепотой» [[источник]](https://t.me/ai_archnadzor/182)

**Проблема:** Линеаризация таблиц в один вектор превращает данные в «семантический шум»

**Решение:** Мульти-векторный индекс (каждой ячейке — свой вектор) + умный роутер

**Результат:** Снижение галлюцинаций в цифрах с **45% до 8%**. Маст-хэв для финтеха и логистики

### 7.6 DSPy 3 и GEPA: Промышленный промпт-инжиниринг [[источник]](https://t.me/ai_archnadzor/184)

**DSPy 3:** LLM как вычислительное устройство. Архитектор описывает Signatures, система генерирует и оптимизирует код промпта

**GEPA (Genetic-Pareto Prompt Optimizer):**
*   Генетические алгоритмы для «скрещивания» лучших промптов
*   Языковая рефлексия — модель анализирует свои ошибки текстом

### 8.1 Semantic Gravity Framework: Физика против галлюцинаций [[источник]](https://t.me/ai_archnadzor/155)

**Проблема:** Синдром «Yes Man» — модель пытается помочь даже с вредоносными запросами или галлюцинирует, чтобы «угодить» пользователю.

**Решение:** Использование физики и геометрии вместо убеждения промптами:

**Геометрия как карта истины:**
1.  **Matryoshka Slicing:** Обрезка векторов до 256 измерений (из 1536) — на 83% быстрее без потери качества.
2.  **SGI (Semantic Grounding Index):** Измеряет, куда «тянет» ответ модели:
    *   Если ответ (R) слишком близок к запросу (Q), но далек от контекста (C) → Sycophancy
    *   SGI = Дистанция(R, Q) / Дистанция(R, C)
    *   Если SGI < 1.0 → Агент «галлюцинирует»

**Физика как движок:**
*   Chain-of-Thought рассматривается как движение частицы в энергетическом поле
*   Высокая энергия = Конфликт между запросом и контекстом
*   Dynamic Beta: «Температура» системы (ниже для сленга, выше для юридических вопросов)

**Петля обратной связи:**
1.  LLM генерирует мысль
2.  Physics Engine считает SGI и энергию
3.  Если энергия слишком высока → мысль отклоняется
4.  Вбрасывается системное сообщение: "Previous thought rejected. Stick strictly to facts."
5.  LLM пробует снова

**Результаты:** 100% Safety Compliance в тестах на 10 индустриях.

## 12. Рекомендации по внедрению ИИ для клиентов

### 12.1 Методология 12-Factor для ИИ [[источник]](AGENTS.md)

1. **Codebase:** Один репозиторий, много развертываний
2. **Dependencies:** Явное объявление всех зависимостей
3. **Config:** Все конфиги через переменные окружения
4. **Backing Services:** Векторные хранилища, LLM API — как подключаемые ресурсы
5. **Build, Release, Run:** Строгое разделение этапов
6. **Processes:** Stateless процессы
7. **Port Binding:** Самодостаточные сервисы
8. **Concurrency:** Масштабирование через процессы
9. **Disposability:** Быстрый старт и graceful shutdown
10. **Dev/Prod Parity:** Минимизация различий сред
11. **Logs:** Логи как поток событий
12. **Admin Processes:** Одноразовые задачи через тот же стек

### 13.1 Федеральный закон № 152-ФЗ «О персональных данных»

**Требования к операторам персональных данных:**

С 1 сентября 2025 года все операторы персональных данных обязаны предоставлять анонимизированные наборы данных в государственную информационную систему по запросу Минцифры [[источник]](https://acsour.com/en/news-and-articles/tpost/2g13ahnab1-mandatory-anonymized-personal-data-shari).

**Ключевые требования:**
- **Локализация данных:** Первичный сбор персональных данных граждан РФ должен осуществляться в базах данных на территории России
- **Аудит соответствия:** Операторы должны пройти аудит соответствия 152-ФЗ
- **Анализ разрывов:** Выявление потенциальных нарушений до аудита
- **Подготовка документов:** Соответствие регуляторным требованиям

**С 1 июля 2025 года:**
- Ужесточение требований к локализации персональных данных
- Запрет обработки персональных данных с использованием иностранных баз данных для:
  - Сбора
  - Записи
  - Систематизации
  - Накопления
  - Хранения
  - Уточнения
  - Извлечения

**Исключения:**
- Запрет не распространяется на трансграничную передачу данных (отдельная операция)
