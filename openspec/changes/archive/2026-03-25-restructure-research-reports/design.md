## Context

**Текущее состояние:**

- В `docs/research/` есть два крупных executive-отчёта:
  - методология внедрения/отчуждения;
  - сайзинг/CapEx/OpEx.
- Оба документа уже содержат пересекающиеся блоки (например, observability, безопасность, отчуждение ИС/кода, описание наработок CMW и отдельные “рыночные дайджесты”), что приводит к:
  - дублированию текста и источников;
  - размытию “канонического” места для деталей;
  - усложнению точечной раздачи материалов стейкхолдерам.

**Ограничения и требования:**

- **Оригинальные отчёты сохраняются неизменными** и служат контрольной базой (ground truth).
- Новый пакет должен быть:
  - удобен для передачи “как один комплект”;
  - пригоден для выборочной раздачи по темам (appendices);
  - с минимальным дублированием и без “лесов” перекрёстных ссылок.
- Внутри новых документов — избегать внутренних путей репозитория как обязательного контекста; перекрёстные ссылки — по человекочитаемым названиям документов.

**Стейкхолдеры:**

- C-level (CEO/CTO/CFO), владельцы продукта, ИБ/комплаенс, архитекторы/LLMOps, предпрод/прод-эксплуатация.

## Goals / Non-Goals

**Goals:**

- Сформировать **пакет из 6 новых документов**: 2 основных + 4 приложения A–D, при этом:
  - **сохранить 100% деталей и смыслов** из оригинальных двух отчётов, разложив их по новым документам (каждый фрагмент — ровно в одном месте как каноническом, либо как короткий “вход” + ссылка на канонику).
  - Уменьшить дублирование, закрепив **право собственности темы** за конкретным документом.
- Сделать “витрину” (Appendix A) **user-friendly**:
  - “какой документ запрашивать по вопросу X”;
  - единый реестр использованных источников (канонический);
  - отдельный список “дополнительных источников из ТЗ” — категоризированный, без оценок надёжности и без статусов использования.
- В каждом документе добавить **короткий раздел навигации** (“если нужно X — см. документ Y”), но держать его на 5–10 строк, чтобы не создавать шум.

**Non-Goals:**

- Не менять фактуру и выводы оригинальных отчётов (контент-реформа, а не пересмотр).
- Не превращать пакет в “энциклопедию”: новые документы могут быть крупными, но должны оставаться executive-ориентированными (SCQA, решения, компромиссы).
- Не внедрять автоматическую “оценку надёжности источников” и не маркировать ссылки статусами использования (во избежание информационного шума).

## Decisions

### 1) Стратегия “2 основных файла + 4 приложения”

**Решение:** делаем **2 основных документа** для стейкхолдеров (методология и экономика) и **4 приложения A–D** как “разворачиваемые” тематические модули.

**Почему:** это компромисс между:

- удобством передачи “пара больших файлов”;
- возможностью раздавать/редактировать темы независимо;
- минимизацией перекрёстных ссылок (приложения — места для детализации, основные — места для решения).

**Альтернатива:** один сверхбольшой документ.

- Минус: тяжело читать/править; сложно “выдать только безопасность” или “только ИС”.

**Альтернатива:** много мелких файлов (8–15).

- Минус: возрастают навигационные издержки и число перекрёстных ссылок, выше риск рассинхронизации.

### 2) Каноничность: “одна тема — один владелец”

**Решение:** вводим правило “**one canonical home**”:

- Экономика и все числа/таблицы/тарифы → **основной отчёт по сайзингу и экономике**.
- Операционная модель, фазы внедрения, методология разработки/внедрения → **основной отчёт по методологии**.
- Отчуждение ИС и кода (KT/IP, лицензии, пакет передачи, чек-листы приёмки) → **Appendix B**.
- Описание наработок CMW (что есть, как компоненты соотносятся, что передаётся) → **Appendix C**.
- Безопасность + комплаенс + observability (как единый управляемый контур контроля/рисков) → **Appendix D**.
- Витрина/реестр/источники/доп. источники → **Appendix A**.

**Почему:** так уменьшается дублирование и исчезает “пинг‑понг” ссылок между основными файлами.

### 3) Appendix A — “витрина”, даже если философски это не приложение

**Решение:** сохраняем обозначение “Appendix A”, но позиционируем как **головной файл навигации** (ведомость документов в дружелюбной форме).

**Почему:** стейкхолдеру важнее “куда идти” и “что попросить”, чем формальная иерархия.

### 4) Политика ссылок и источников

**Решение:**

- **Единый канонический реестр источников** — в Appendix A.
- В каждом документе:
  - оставляем inline-ссылки там, где они необходимы для доказательности ключевых тезисов;
  - раздел `## Источники` допускается, но должен ссылаться на Appendix A как на “полный реестр” (чтобы не дублировать длинные списки).
- “Дополнительные источники из ТЗ” — отдельным блоком в Appendix A, **категоризированно**, без оценок надёжности и без статусов использования.

**Почему:** снимаем дублирование, но сохраняем проверяемость ключевых утверждений.

### 5) Механизм гарантии “ничего не потеряли”

**Решение:** ввести процесс “traceability” на уровне редакторской процедуры:

- “Карта переноса” (mapping) из оригиналов → новые документы:
  - таблица “раздел/тема/смысловой блок → целевой документ → целевой раздел”.
- Итоговая проверка:
  - для каждого раздела оригиналов есть назначение “куда перенесено”;
  - большие смысловые блоки встречаются **ровно один раз** в новых документах (кроме коротких executive-выжимок, которые разрешены как ввод).

**Почему:** это единственный надёжный способ соблюсти требование “keep all details only place across the new docs”.

## Risks / Trade-offs

- **[Риск] Рост объёма “витрины” из‑за реестра источников** → **Митигация:** разделить Appendix A на “Навигация (коротко)” и “Реестры (длинно)”, где навигация остаётся на 1–2 экрана.
- **[Риск] Скрытое дублирование (тема всплывает в 2 местах)** → **Митигация:** правило “owner of topic” + “карта переноса” + финальная валидация.
- **[Риск] Слишком много перекрёстных ссылок** → **Митигация:** ссылки только “вниз” (из основных в приложения), избегать ссылок “по кругу”.
- **[Риск] Appendix D слишком широк (безопасность+комплаенс+observability)** → **Митигация:** внутри D сделать жёсткую структуру и выделить “минимальный контур” vs “углубление”; при перегрузе — допускается разбиение D на D1/D2 в будущем, но не в первой версии пакета.

## Migration Plan

- Создать 6 новых файлов в `docs/research/` с неймингом по текущей конвенции (`YYYYMMDD-...-ru.md`), не трогая оригинальные `20260323-...`.
- Перенести контент из оригиналов в новые документы по “карте переноса”.
- Обновить перекрёстные ссылки и навигационные блоки.
- Выполнить финальную сверку “coverage” (по карте переноса) и устранить повторы.
