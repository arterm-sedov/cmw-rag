## 3. Пошаговый план валидации и улучшения

### Этап 1. Проверка когерентности терминологии (День 1)

#### Контрольная точка 1.1: Единообразие терминов

**Действия:**

1. **Проверить все документы на использование терминов:**
   - CapEx / OpEx / TCO — единое написание
   - KT / IP / BOT — единая расшифровка при первом упоминании
   - RAG / LLM / GenAI — единый перевод при первом упоминании
   - ПДн / 152-ФЗ — единое написание

2. **Проверить якоря валюты:**
   - Приложение A: `#app_a_fx_policy` — канонический блок
   - C-level резюме (методология): `#exec_method_currency` — ссылка на A
   - C-level резюме (сайзинг): `#exec_sizing_currency` — ссылка на A
   - Основные отчёты: ссылки на `#app_a_fx_policy`

3. **Проверить числовые значения:**
   - Все USD-суммы пересчитаны в RUB по курсу 85
   - Формат чисел: пробел как разделитель разрядов, запятая как десятичный разделитель

**Критерий приёмки:** Все термины и константы согласованы, противоречия устранены.

---

### Этап 2. Валидация бизнес-посылов (День 1–2)

#### Контрольная точка 2.1: Соответствие бизнес-целям

**Действия:**

1. **Проверить каждый документ на соответствие целям:**
   - Трансфер экспертизы: есть ли чёткий комплект отчуждения?
   - Коммерциализация: есть ли обоснование ROI/TCO?
   - Обеспечение продаж: есть ли матрица решений для C-Level?

2. **Проверить связку «продукт — ценность — доказательство»:**
   - Что продаём: внедрение + передача экспертизы
   - Ценность для клиента: суверенитет, владение, измеримый ROI
   - Доказательство: публичные источники + инженерная практика Comindware

3. **Проверить разделение «Comindware» vs «открытые источники»:**
   - Явно указано, что практика Comindware — в репозиториях
   - Явно указано, что открытые источники — с атрибуцией и оговорками

**Критерий приёмки:** Каждый документ чётко связан с бизнес-целями, нет «оторванных» разделов.

### Этап 3. Проверка непротиворечивости цифр (День 2)

#### Контрольная точка 3.1: Согласованность количественных данных

**Действия:**

1. **Создать реестр всех количественных утверждений:**
   - Тарифы (₽/млн токенов)
   - CapEx/OpEx диапазоны
   - Рыночные доли и прогнозы
   - KPI и целевые метрики

2. **Проверить дублирование цифр:**
   - Если цифра появляется в нескольких документах — она одинакова
   - Если есть расхождение — явно указана причина или ссылка на источник

3. **Проверить особые случаи:**
   - Две линии ~43% в опросе CMO (галлюцинации vs утечка) — не смешивать
   - Выборка OpenAI enterprise — явно указана как выборка, не норма для РФ
   - Прогнозы IMARC vs сегментные оценки — разные методологии, не суммировать

**Критерий приёмки:** Все цифры согласованы, противоречия устранены или явно помечены.

### Этап 5. Проверка комплаенса и правовых аспектов (День 3)

#### Контрольная точка 5.1: 152-ФЗ и суверенный контур

**Действия:**

1. **Проверить все упоминания 152-ФЗ:**
   - Единая формулировка: «152-ФЗ „О персональных данных“»
   - Ссылка на Приказ Роскомнадзора № 140 (обезличивание)

2. **Проверить разделение контуров:**
   - Промышленный контур (on-prem / облако РФ)
   - Теневой GenAI (маркетинг / личные аккаунты)
   - Явные риски утечки через неучтённые каналы

3. **Проверить оговорки по зарубежным источникам:**
   - OpenAI, Anthropic, Meta — явно указаны как зарубежные
   - EU AI Act — только как сравнительный контекст
   - NIST AI RMF — методологический якорь, не норма РФ

**Критерий приёмки:** Все правовые аспекты согласованы, оговорки на месте.

### Этап 6. Проверка стиля и форматирования (День 3–4)

#### Контрольная точка 6.1: Единообразие оформления

**Действия:**

1. **Проверить заголовки:**
   - Нет нумерованных заголовков в финальных отчётах
   - Нет гиперссылок внутри заголовков
   - Якоря в формате `{: #anchor_slug }`

2. **Проверить admonitions (callout-блоки):**
   - Используются `!!! tip`, `!!! warning`, `!!! note` вместо `**Рекомендация:**`
   - Формат: `!!! tip "Заголовок"` с отступом 4 пробела

3. **Проверить списки:**
   - Пустая строка перед каждым списком
   - Маркер `-` (не `*`)
   - Нет orphan-пробелов на пустых строках

4. **Проверить кавычки и тире:**
   - Русские кавычки `«»` для русского текста
   - Em dash `—` для тире
   - En dash `–` для диапазонов

**Критерий приёмки:** Единый стиль оформления во всех документах.

### Этап 7. Финальная валидация C-Level резюме (День 4)

#### Контрольная точка 7.1: Автономность резюме

**Действия:**

1. **Проверить отсутствие внутренних путей:**
   - Нет `.md`-файлов в теле резюме
   - Нет ссылок на репозитории
   - Связь через названия документов

2. **Проверить SCQA-структуру:**
   - Ситуация: контекст рынка и РФ
   - Осложнение: ключевой вызов
   - Вопрос: основной вопрос для решения
   - Ответ: конкретные рекомендации

3. **Проверить план действий 30/60/90:**
   - Конкретные действия
   - Измеримые критерии
   - Связь с основным отчётом

**Критерий приёмки:** Резюме полностью автономны, готовы к передаче C-Level.

## 6. Глоссарий российской бизнес-терминологии

### 6.1 Трансфер технологий и экспертизы

| Термин | Определение | Английский аналог |
|:---|:---|:---|
| **Трансфер технологий** | Передача технологий, знаний и компетенций от одной организации другой | Technology Transfer |
| **Передача экспертизы** | Процесс передачи специализированных знаний и навыков заказчику | Expertise Transfer / Knowledge Transfer (KT) |
| **Отчуждение ИС** | Полная передача исключительных прав на интеллектуальную собственность | IP Alienation / IP Assignment |
| **Секрет производства (ноу-хау)** | Конфиденциальная информация, составляющая коммерческую тайну | Know-how / Trade Secret |
| **Лицензионный договор** | Предоставление права использования ИС без передачи права собственности | License Agreement |
| **Договор об отчуждении исключительного права** | Полная передача прав на результат интеллектуальной деятельности | IP Assignment Agreement |

### 6.2 Модели взаимодействия

| Термин | Определение | Английский аналог |
|:---|:---|:---|
| **BOT (Построение — Эксплуатация — Передача)** | Модель, при которой интегратор создаёт, эксплуатирует и передаёт систему заказчику | Build–Operate–Transfer |
| **Управляемый сервис** | Модель, при которой поставщик полностью управляет системой | Managed Service |
| **Совместная разработка** | Модель сотрудничества команд заказчика и интегратора | Co-development / Joint Development |
| **Создание и передача** | Модель разработки «под ключ» с последующей передачей | Create–Transfer |

### 6.3 Экономика и финансы

| Термин | Определение | Английский аналог |
|:---|:---|:---|
| **Капитальные затраты (CapEx)** | Инвестиции в долгосрочные активы (оборудование, инфраструктура) | Capital Expenditure |
| **Операционные расходы (OpEx)** | Текущие расходы на эксплуатацию и сопровождение | Operating Expense |
| **Совокупная стоимость владения (TCO)** | Полная стоимость актива за весь жизненный цикл | Total Cost of Ownership |
| **Юнит-экономика** | Расчёт экономической эффективности на единицу продукта/услуги | Unit Economics |
| **Точка безубыточности** | Объём продаж, при котором доходы равны расходам | Break-even Point |
| **Чувствительность модели** | Зависимость результатов от изменения входных параметров | Model Sensitivity |

### 6.4 Организационная модель

| Термин | Определение | Английский аналог |
|:---|:---|:---|
| **Целевая операционная модель (TOM)** | Желаемое состояние процессов, ролей и технологий | Target Operating Model |
| **Центр компетенций** | Подразделение, аккумулирующее экспертизу в определённой области | Center of Excellence (CoE) |
| **Федеративная модель** | Распределённая модель управления с сильным центром | Federated Model |
| **Матрица ролей и ответственности** | Распределение обязанностей между участниками | RACI Matrix |

### 6.5 Комплаенс и безопасность

| Термин | Определение | Английский аналог |
|:---|:---|:---|
| **Персональные данные (ПДн)** | Информация, относящаяся к физическому лицу | Personal Data |
| **Обезличивание данных** | Удаление или маскирование идентификаторов личности | Data Anonymization |
| **Оператор персональных данных** | Организация, определяющая цели обработки ПДн | Data Controller |
| **Обработчик персональных данных** | Организация, осуществляющая обработку ПДн по поручению оператора | Data Processor |
| **Субъект персональных данных** | Физическое лицо, чьи данные обрабатываются | Data Subject |
| **Трансграничная передача данных** | Передача данных за пределы юрисдикции | Cross-border Data Transfer |

### 6.6 Технологии ИИ

| Термин | Определение | Английский аналог |
|:---|:---|:---|
| **Генеративный ИИ (GenAI)** | ИИ, способный создавать новый контент | Generative AI |
| **RAG (Генерация с дополненной выборкой)** | Метод, комбинирующий поиск и генерацию | Retrieval-Augmented Generation |
| **LLM (Большая языковая модель)** | Модель, обученная на больших текстовых корпусах | Large Language Model |
